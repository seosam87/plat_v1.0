# Roadmap v3.0: Workflows & Automation — SEO Management Platform

## Концепция

Каждая фаза = рабочий workflow или инструмент автоматизации. Фокус на реальных сценариях SEO-специалиста, работающего с 20-100 сайтами.

## Фазы

- [ ] **Phase 1: Yandex Metrika** — Трафик по страницам, сравнение периодов, новый трафик
- [ ] **Phase 2: Content Audit Engine** — WP-страницы с фильтрами, чеклист по типу, CTA, schema
- [ ] **Phase 3: Change Monitoring** — Алерты при изменениях на сайтах, еженедельный дайджест
- [ ] **Phase 4: Analytics Workspace** — Фильтр→temp group→позиции→SERP→конкурент→сравнение
- [ ] **Phase 5: Content Gap Analysis** — Ключи конкурентов, которых нет у нас → контент-план
- [ ] **Phase 6: Site Architecture** — SF import, дерево сайта, pillar-service-article
- [ ] **Phase 7: Bulk Operations Hub** — Массовые операции над ключами и страницами
- [ ] **Phase 8: Cannibalization Resolver** — Action plans для каннибализации
- [ ] **Phase 9: Intent Auto-Detect** — Парсинг Яндекс SERP → определение коммерческости
- [ ] **Phase 10: Traffic Analysis & Bot Detection** — Анализ подлива трафика, точки входа, боты

---

## Phase Details

### Phase 1: Yandex Metrika
**⚠ ОБЯЗАТЕЛЬНО: обсудить с пользователем перед началом — у него много идей по парсингу**

**Модуль:** `metrika` — новый сервис + роутер + UI

- Подключение через API Метрики (OAuth-токен)
- Органический трафик по страницам
- Сравнение периодов (этот месяц vs прошлый)
- Новый трафик — страницы, которые начали получать визиты
- Визиты, глубина просмотра, время на сайте
- Интеграция в Dashboard и Site Overview

---

### Phase 2: Content Audit Engine
**Модуль:** расширение Pipeline + новый UI «Content Audit»

**Общий workflow:**
1. Получить список WP-страниц с фильтрами (title, дата, тип)
2. По каждой странице — чеклист проверок (зависит от типа)
3. Исправить проблемы → approve → push

**Информационные страницы (блог/статья):**
- TOC — обязательно
- Блок автора — проверить наличие
- Похожие статьи (related posts) — проверить наличие
- Schema.org: Article, BlogPosting
- Внутренние ссылки

**Коммерческие страницы (услуга/товар):**
- CTA-блок — проверить, если нет → вставить в конец (HTML генерируется вне системы)
- Schema.org: Service, Product, LocalBusiness, FAQ
- Внутренние ссылки

**Что создать:**
1. UI для WP-страниц с фильтрами (search, date range, page type)
2. Расширяемый чеклист проверок
3. Type-aware pipeline (разные действия для info vs commercial)
4. CTA injection endpoint

---

### Phase 3: Change Monitoring
**Модуль:** расширение Crawl + Telegram/Email notifications

- Автоматическое отслеживание изменений между краулами
- Алерты в Telegram при:
  - Изменение title/H1/meta description
  - Появление noindex на странице
  - Пропажа страниц (404)
  - Изменение canonical
  - Удаление schema.org
- Еженедельный дайджест: сводка изменений по всем сайтам
- Настраиваемые правила: какие изменения важны, какие игнорировать

---

### Phase 4: Analytics Workspace
**Модуль:** новый UI «Workspace» — сквозной workflow аналитика

1. Расширенные фильтры ключевых слов (частотность, регион, группа, intent, позиция)
2. Сохранение выборки во временную тест-группу
3. Проверка позиций по группе (не per-site, а per-group)
4. SERP-парсинг по группе
5. Обнаружение топового конкурента из SERP
6. Сравнение позиций конкурента
7. Сравнение SEO-полей, размера, структуры страниц

---

### Phase 5: Content Gap Analysis
**Модуль:** расширение Competitors

- Пересечение SERP-данных конкурентов с нашими ключами
- Ключи, по которым конкуренты в TOP-10, а мы нет → «упущенные темы»
- Автоматическое создание предложений в контент-план
- Группировка gap-ключей по тематике
- Оценка потенциала: частотность × вероятность попадания в TOP

---

### Phase 6: Site Architecture
**Модуль:** расширение Crawl + SF Import

- Импорт данных Screaming Frog: SEO-поля, ссылки, структура
- Построение дерева сайта (визуализация URL-иерархии)
- Сравнение с sitemap.xml (orphan pages, missing from sitemap)
- Идеальная структура pillar-service-article: специалист задаёт шаблон, система сравнивает
- Отслеживание изменений ссылочной структуры между краулами

---

### Phase 7: Bulk Operations Hub
**Модуль:** расширение Keywords + UI

- Массовый перенос ключей между группами/кластерами
- Массовое назначение target_url (по паттерну или вручную)
- Массовое удаление ключей по фильтру
- Export с применёнными фильтрами (CSV/Excel)
- Import с merge в существующие данные
- Select all + batch action на любой таблице с ключами

---

### Phase 8: Cannibalization Resolver
**Модуль:** расширение Clusters + Tasks

- Не только обнаружение каннибализации, но предложение решения:
  - Объединить страницы (merge content)
  - Настроить canonical (указать главную)
  - Настроить redirect (301)
  - Разнести ключи по разным страницам
- Создание задачи с конкретным action plan
- Отслеживание: решена ли каннибализация после действий

---

### Phase 9: Intent Auto-Detect
**Модуль:** расширение Clusters + SERP Parser

- Парсинг Яндекс SERP по ключу + регион
- Анализ выдачи: если в TOP-10 преобладают коммерческие страницы → коммерческий intent
- Полуавтомат: система предлагает, специалист подтверждает
- Пакетная обработка: определить intent для всех некластеризованных ключей
- Обновление intent кластера на основе результатов

---

### Phase 10: Traffic Analysis & Bot Detection
**Модуль:** новый `traffic_analysis` — анализ подлива трафика и ботов

**Источники данных (три варианта, комбинируемые):**
1. **Логи сервера** — загрузка access.log, парсинг IP/UA/URL/timestamp
2. **Yandex Metrika** — выгрузка визитов с детализацией (referer, geo, device)
3. **WP-плагин** (опционально) — собирает данные на стороне сайта, отдаёт через REST API

**Что анализировать:**
- Точки входа трафика (landing pages)
- Источники (referer, UTM)
- Паттерны ботов: аномальная частота, подозрительные UA, нетипичный geo
- Разделение: свои боты / чужие боты / бесплатное «продвижение» / органика
- Timeline: когда начался подлив, когда закончился
- Влияние на поведенческие метрики (если Метрика подключена)

**UI:**
- Дашборд трафика: timeline визитов по типам
- Алерты при аномальных всплесках
- Фильтрация по IP/UA/referer
- Сравнение «до подлива» vs «во время» vs «после»
