# Phase 999.5 Context — Repo ↔ Test-Prod Sync Tooling

## Motivation

Two deployment-drift incidents in one work session between `/projects/test/` (dev git checkout) and `/opt/seo-platform/` (local test-prod checkout running via `docker-compose.prod.yml` in the same machine):

1. **Plan 15.1-03:** `opportunities_gaps.html` fix was committed to repo (f1259c1) but `/opt/seo-platform/app/templates/...` was stale. `data.items` regression test only passed after manual sync.
2. **Debug phase-15.1-deferred-routes Group A:** `opportunities_{cannibal,losses}.html` same story — fix in repo, stale in test-prod.

Root cause diagnosis (from discussion 2026-04-07):

- `/opt/seo-platform/` is a **local test-prod** on the same machine (NOT the real VPS, NOT a bind-mount of dev). It's an independent git checkout with its own docker-compose stack that simulates production.
- It was sitting on commit `f5f38a1` (Phase 15 era, several phases behind current dev `75fdf82`) — **not intentional release pinning**, just "forgot to `git pull`".
- Current uncommitted changes in `/opt/seo-platform/app/routers/{audit,monitoring,metrika}.py` are the Group B router-ordering fixes applied manually during the debug session — they will be lost on the next `git pull` unless explicitly committed there.
- The real production server is elsewhere (separate VPS). `/opt/seo-platform/` exists specifically to let the user run smoke tests "in conditions close to prod" before deploying to the real VPS.

**User's stated workflow (confirmed during discussion):**
> "I want to keep manual control over when to update test-prod (`git pull` in `/opt/seo-platform/`), but I want a command that runs the tests against it and tells me which commit I'm actually testing. And if I haven't synced in a while, I want to be reminded."

This phase delivers that tooling. It does NOT automate the sync itself — the user keeps the `git pull` step under their control.

## Scope

**In scope:**
- A single shell script `./scripts/test-my-platform.sh` that runs the full test suite (`pytest tests/`) inside the test-prod docker stack and prints the commit hash under test
- A divergence-warning mechanism that fires specifically when the user asks Claude to push changes to git — warns if `/opt/seo-platform/` HEAD differs from `/projects/test/` HEAD
- One-time commit of the currently-uncommitted manual fixes in `/opt/seo-platform/app/routers/...` as a dedicated "release sync" commit there
- Short docs section in `tests/README.md` explaining the dev / test-prod split and the workflow

**Out of scope:**
- Automating `git pull` itself (user wants manual control)
- Release tags, CHANGELOG, promotion scripts (premature for a solo-dev + Claude workflow — can become phase 999.x later if needed)
- Anything touching the real production VPS
- Any change to dev-tree docker-compose.yml (the Phase 999.4 `tests/` bind-mount already landed)
- Detecting or fixing drift at the _working tree_ level (untracked/uncommitted changes in test-prod beyond the one-time initial commit) — covered in deferred

## Decisions (locked)

### D-01: Test command — `./scripts/test-my-platform.sh`

Single bash script in the repo at `scripts/test-my-platform.sh`. Responsibilities:

1. `cd /opt/seo-platform` and capture `DEV_HEAD=$(cd /projects/test && git rev-parse HEAD)`, `PROD_HEAD=$(git rev-parse HEAD)`, `PROD_DIRTY=$(git status --porcelain | wc -l)`
2. Print a 3-line header:
   ```
   Dev HEAD:      {DEV_HEAD}  (/projects/test)
   Test-prod HEAD: {PROD_HEAD} (/opt/seo-platform)  [behind by N commits]  [DIRTY: M files]
   Running: pytest tests/ inside seo-platform-api-1
   ```
3. If `DEV_HEAD != PROD_HEAD` → print a warning line but DO NOT block. No interactive `y/n` prompt — the user explicitly wants the script to just run.
4. Exec `docker compose -f /opt/seo-platform/docker-compose.prod.yml exec -T api pytest tests/` and stream output
5. On completion, print a footer with the same HEAD info + the pytest exit code

The script is idempotent, read-only against both trees, and does nothing on failure beyond returning the pytest exit code.

**Why:** User wants a single command that (a) runs the full suite against test-prod, (b) makes the commit under test unambiguous, (c) doesn't try to be clever about fixing drift. All three requirements satisfied in one file.

### D-02: Run full `pytest tests/`, not just smoke

Per user preference: the script runs the complete `pytest tests/` suite, not just `tests/test_ui_smoke.py`. Takes longer (minutes vs seconds) but is what the user asked for when answering question B during discussion.

**Why:** The user's mental model is "test-my-platform = full sanity check against real-like environment". Speed is secondary to completeness.

### D-03: Divergence warning fires on git push intent, not on every prompt

The warning is NOT a PostToolUse hook firing on every Claude response, NOT a SessionStart banner. It fires **specifically when the user asks Claude to push changes to git** (e.g., "запушь", "push to git", "commit and push", etc.).

Mechanism: Claude Code PreToolUse hook on `Bash` tool calls matching `git push` patterns, OR a check Claude performs itself at the moment of any tool call that runs `git push`. Both options are on the table — the planner can decide which is cleaner.

**What the warning says (exact text, Russian per user preference):**
```
⚠ Внимание: test-prod (/opt/seo-platform) отстаёт от dev на {N} коммитов.
  Dev HEAD:      {DEV_HEAD_short}  — {DEV_HEAD_subject}
  Test-prod HEAD: {PROD_HEAD_short} — {PROD_HEAD_subject}

  Перед push'ем есть смысл прогнать ./scripts/test-my-platform.sh,
  чтобы тесты проверили код в условиях, близких к проду.

  Продолжить push без запуска тестов? [y/N]
```

**If user confirms (y):** push proceeds normally
**If user declines (N or anything else):** push aborts, Claude explains next step

Threshold per user answer C: **ANY divergence ≥ 1 commit triggers the warning**. No grace period, no time-based threshold. If user doesn't want to be bothered by a 1-commit divergence, they can just press `y`.

**Why:** The user explicitly wants to be reminded whenever they're about to push a batch of changes and test-prod isn't in sync. They picked "any divergence" over "5+ commits" to avoid missing anything. Confirmation prompt lets them opt out in-context without code changes.

### D-04: One-time commit of current test-prod working-tree changes

Before any sync tooling is built, the current uncommitted state in `/opt/seo-platform/` must be preserved:

1. `cd /opt/seo-platform`
2. `git status` to enumerate exactly which files differ from `f5f38a1`
3. Commit those changes with a dedicated message:
   ```
   fix(opt-seo-platform): apply router ordering + metrika widget fixes from debug phase-15.1-deferred-routes

   Ported from dev commits 471a88d + 178c2df. This commit exists only in
   the /opt/seo-platform test-prod tree — dev tree has the canonical
   fixes. Will be rebased out on next git pull to sync with dev master.
   ```
4. Do NOT force a merge with dev master in this phase — that's the user's manual decision.

**Why:** Losing the manual Group B fixes on the next `git pull` is a landmine. This commit ensures a clean recovery point. The message explicitly flags the commit as "temporary, will be rebased" so future Claude sessions understand it's not a long-lived branch point.

### D-05: Documentation — short section in tests/README.md

Add a "## Test-Prod vs Dev Tree" section to `tests/README.md`:

- One paragraph explaining the dev / test-prod split (this machine has two checkouts)
- Commands cheat sheet:
  - Update test-prod manually: `cd /opt/seo-platform && git pull`
  - Run tests against test-prod: `./scripts/test-my-platform.sh`
  - Check divergence: `./scripts/test-my-platform.sh --check` (just prints heads, skips pytest)
- Explicit note: "Real production VPS is separate. test-prod is local only."

**Why:** Documentation for future-you + future-Claude-sessions so the mental model doesn't have to be re-derived. Minimal scope — not a deployment guide.

### D-06: `--check` flag on the script for fast divergence peek

`./scripts/test-my-platform.sh --check` prints only the divergence info (the 3-line header from D-01) and exits 0 if HEADs match, exits 1 if they diverge. Skips the pytest run entirely. Enables the push-hook in D-03 to reuse the same script rather than reimplementing the git comparison logic.

**Why:** Single source of truth for "how do we check divergence" — script + hook call the same code.

### D-07: Hook implementation via Claude Code settings.json

The D-03 divergence warning is implemented as a Claude Code PreToolUse hook in `.claude/settings.json`, matching `Bash` tool calls whose command contains `git push`. The hook runs `./scripts/test-my-platform.sh --check`; if it exits non-zero, the hook returns a blocking response with the warning text and the confirmation prompt.

Fallback: if hook mechanism isn't viable (settings.json limits, JSON hooks can't do interactive prompts), use a shell wrapper: the hook simply injects an advisory message into Claude's context and Claude asks the user the confirmation question conversationally before proceeding.

**Why:** Hooks are the "right" place for this in Claude Code architecture. Conversational fallback ensures the feature ships even if hooks prove limiting.

### D-08: Do NOT modify docker-compose.yml in either tree

Explicit: this phase does not touch either docker-compose.yml file. The dev tree's `tests/` bind-mount was added in phase 999.4 quick fix. The `/opt/seo-platform/docker-compose.prod.yml` is prod-shaped on purpose — we do NOT add bind mounts to make it mirror dev, because then it stops being "test-prod" and becomes "another dev".

**Why:** Scope discipline + preserving the test-prod's value proposition (it's supposed to be different from dev to catch bind-mount-only illusions).

## Risks / pitfalls

1. **Claude Code hook API may not support interactive confirmation.** If the PreToolUse hook mechanism can only return allow/block without asking the user a question mid-flow, fall back to conversational path (hook injects an advisory, Claude asks the user directly before calling `git push`). Planner must verify hook capabilities before committing to one approach.
2. **The test-prod container may not be running when script is invoked.** `docker compose exec` fails if container is down. Script should detect this and fail fast with a clear message: "test-prod container is not running. Start it with `cd /opt/seo-platform && docker compose -f docker-compose.prod.yml up -d`".
3. **`/opt/seo-platform/` may not exist on other machines.** This phase is specifically for the user's current setup. If the script is run on a machine without `/opt/seo-platform/`, it should exit with a clear "test-prod tree not found at /opt/seo-platform/ — this script is a local-machine convenience, not a portable tool".
4. **Commit per D-04 introduces a divergence between /opt/seo-platform/ and origin/master.** This is intentional but surprising if you `git pull` later — there will be a merge conflict or a rebase needed. Document this explicitly in the commit message and in README.
5. **The push hook fires on ALL git push operations, including pushing to feature branches.** User may not want to be warned when pushing an experimental branch. Consider: only fire the warning for pushes to `master` / `main`. Planner should discuss this during planning if it feels over-scoped here.

## Deferred

- Auto-detection of working-tree drift in test-prod beyond the one-time D-04 commit (if someone hand-edits files in `/opt/seo-platform/` after this phase, no tooling catches it — user accepts that risk)
- Release workflow (tags, CHANGELOG, deploy-to-VPS automation) — explicitly out of scope, may become a later phase
- CI workflow integration (push-warning in GitHub Actions) — local-only for now
- Time-based divergence staleness ("haven't synced in 4h") — user picked "any divergence" over time-based in question C
