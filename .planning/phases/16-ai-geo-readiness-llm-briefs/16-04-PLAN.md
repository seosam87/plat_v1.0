    Create `app/routers/llm_briefs.py` with prefix `/briefs` and dependency `get_current_user`:

    - `POST /briefs/{brief_id}/llm-enhance` →
      1. Load brief, verify ownership.
      2. If `not current_user.has_anthropic_key` → return 400 with HTMX-friendly error partial "Настройте Anthropic API key в Профиле".
      3. Check Redis circuit: if `is_circuit_open(redis, current_user.id)` → return 429 partial "LLM временно недоступен (3 ошибки подряд). Попробуйте через 15 минут."
      4. Create `LLMBriefJob(user_id=current_user.id, brief_id=brief_id, status='pending')`, commit to get id.
      5. Dispatch `generate_llm_brief_enhancement.delay(job.id)`.
      6. Return HTMX partial that immediately starts polling: `<div hx-get="/briefs/llm-jobs/{job.id}" hx-trigger="load, every 2s" hx-swap="outerHTML">генерация…</div>`

    - `GET /briefs/llm-jobs/{job_id}` →
      1. Load LLMBriefJob, verify ownership.
      2. If status in ('pending','running'): return same polling div with text "генерация… ({elapsed}s)".
      3. If status='done': return preview partial with 3 collapsible sections (expanded_sections, faq_block, title_variants+meta_variants) + Accept button (POST `/briefs/llm-jobs/{job_id}/accept`) + Regenerate button (POST `/briefs/{brief_id}/llm-enhance`). Stop polling by NOT including hx-trigger.
      4. If status='failed': return red error block with `job.error_message` + Regenerate button. No polling.

    - `POST /briefs/llm-jobs/{job_id}/accept` →
      1. Load job, verify done + ownership.
      2. Render the merged view: return a partial showing the brief WITH the AI sections inlined (this is a UI merge — do NOT mutate the underlying ContentBrief row; LLM-03 requires the template brief itself remains untouched. The "merge" is a UI overlay only). Mark job with a new column? — NO: just return the merged HTML; if the user reloads, they re-trigger.
      3. Alternative if simpler: store `job.status='accepted'` and brief detail view checks for the latest accepted job for that brief and inlines its output_json. Choose whichever is cleaner during implementation; document in SUMMARY.
