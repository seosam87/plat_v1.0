# Phase 19.2: Interactive Tour Player — Context

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Frontend JS overlay (`app/static/js/tour.js` — Shepherd.js wrapper) that consumes the same `scenarios/*.yaml` files shipped in Phase 19.1 and plays them as interactive admin onboarding tours over the live application. Admin-only "Show tour" button appears on pages that have a registered tour. One source of truth: same YAML schema, same `scenarios/` directory, two runners (pytest in 19.1, browser overlay in 19.2).

**In scope:**
- `app/static/js/tour.js` — Shepherd.js-based player loaded from base.html
- New optional top-level `tour:` block in scenario YAML schema (Pydantic, backward-compatible)
- FastAPI endpoints `GET /api/tours/` (list, filtered by page) and `GET /api/tours/{name}` (single tour JSON)
- Tour-only scenario YAML files in the existing `scenarios/` directory using the reserved `say`/`highlight`/`wait_for_click` ops + optional `open` for navigation
- Floating admin-only "Show tour" button rendered conditionally on pages where at least one tour matches `location.pathname`

**Out of scope (deferred):**
- Migrating existing test scenarios (`01-suggest-to-results.yaml`, `02-site-form-submit.yaml`) into tours
- Multi-language tour text (Russian only for now, mirrors existing UI)
- Tour analytics / completion tracking
- Per-user tour-seen state (every admin can replay any tour any time)
- Non-admin user tours

</domain>

<decisions>
## Implementation Decisions

### Tour Engine
- **D-01:** Use **Shepherd.js loaded via CDN** in `app/templates/base.html`, mirroring the existing CDN pattern (htmx, tailwind, flatpickr already loaded this way). No bundler, no npm. Pin a specific version (e.g. `shepherd.js@14.x`). `app/static/js/tour.js` is a thin wrapper that translates parsed scenario JSON into Shepherd `Tour` + `Step` objects.
- **Why:** Shepherd already solves popper positioning, modal overlay, keyboard nav, prev/next, scroll-into-view — reimplementing that as a custom 200-LOC `tour.js` would just be a worse Shepherd. Fits the project's "load libs from CDN, no build step" convention.

### Step Op Handling (hybrid)
- **D-02:** Tour player handles ops as follows:
  - **Tour-native (execute):** `say` → render Shepherd step text in tooltip; `highlight` → attach Shepherd step to `target` element; `wait_for_click` → advance the tour when the user clicks `target` (instead of clicking "Next").
  - **Navigation (execute):** `open` → `window.location.assign(url)` and resume the tour after page load (state persisted in `sessionStorage` so the tour survives navigation).
  - **Test-only (skip with warning):** `click`, `fill`, `wait_for`, `expect_text`, `expect_status` → log `console.warn("[tour] skipping test-only op: <op>")` and move to the next step.
- **Why:** Mirrors how 19.1 skips reserved tour ops with a warning — symmetric contract. Auto-executing `open` lets a tour start from any page; auto-filling forms or auto-clicking buttons would be confusing UX (the user wants to learn, not watch a robot).

### Tour↔Page Binding
- **D-03:** Add a new optional top-level `tour:` block to the scenario YAML schema:
  ```yaml
  tour:
    entry_url: /ui/keyword-suggest/    # required if `tour:` present
    title: "Подбор ключей"             # required
    intro: "Покажу как собрать подсказки за 30 секунд"  # optional
  ```
  Pydantic schema (`tests/fixtures/scenario_runner/schema.py`) gets a new optional `tour: Optional[TourMeta] = None` field. Scenarios without `tour:` are invisible to the tour player; scenarios with `tour:` are advertised by the API and matched to pages via `entry_url` prefix against `location.pathname`.
- **Why:** One file, two contracts. No new directories, no filename magic, no DB table. Backward-compatible — existing 19.1 test scenarios remain valid without modification. The Pydantic discriminated union for steps is untouched.

### Authoring Model (tour-only files)
- **D-04:** Tours are authored as **separate YAML files in the existing `scenarios/` directory**, using only tour-native ops (`say`, `highlight`, `wait_for_click`) plus optional `open` for navigation. They contain a `tour:` block; they do NOT contain `click`/`fill`/`expect_*` ops. Existing test scenarios (`01-suggest-to-results.yaml`, `02-site-form-submit.yaml`) are NOT migrated into tours — they stay test-only.
- **D-04a:** Phase 19.2 ships **at least one real tour** as a phase-gate, e.g. `scenarios/tours/keyword-suggest-walkthrough.yaml` (or flat in `scenarios/` — planner decides). Planner picks the most valuable first tour from the v2.1 onboarding domain.
- **Why:** Test scenarios fill forms with "smoke seed" — useless and confusing as user-facing tours. Splitting authoring keeps each file's intent clear. Same schema + same directory means the contract stays unified; only the `tour:` block flips a file from "invisible to player" to "advertised by player".

### Backend API
- **D-05:** Add a thin FastAPI router (likely `app/web/routes/tours.py`) with two endpoints:
  - `GET /api/tours/?page=<pathname>` → returns a JSON array of tours whose `tour.entry_url` is a prefix of `page`. Each item: `{name, title, intro}`.
  - `GET /api/tours/{name}` → returns the full parsed scenario as JSON (steps array + tour metadata). Server-side validation reuses `tests/fixtures/scenario_runner/schema.py` (or a thin shim that imports the Pydantic model — planner decides whether to move the schema out of `tests/` to make it importable from `app/`).
  - Both endpoints are admin-gated (existing auth dependency).
- **Why:** Server already has Pydantic + PyYAML. Browser stays lean (no `js-yaml` CDN, ~15KB saved). One round-trip per page load gives the player everything it needs. The schema-import question (move out of `tests/`?) is flagged for the planner — see Open Questions.

### "Show Tour" Button
- **D-06:** A small JS snippet in `base.html` (or `tour.js` itself) runs on every page for admin users:
  1. `fetch('/api/tours/?page=' + encodeURIComponent(location.pathname))`
  2. If the response is non-empty, render a **floating button bottom-right** ("Показать тур" with a 🎓 or similar icon).
  3. Click → fetch `/api/tours/{name}`, instantiate Shepherd tour, start.
  4. If multiple tours match the page → show a small popover with a list (rare in v2.1; planner can defer multi-tour UI if only one tour ships).
- Non-admin users: button is never rendered (server returns 403 or empty list — planner picks the cleaner option).
- **Why:** Zero per-template work — no need to add `{% block tour_button %}` to every page. UI stays clean on pages without tours. Admin gating via the existing auth dependency, no new permission system.

### Claude's Discretion
- **CD-01:** Exact Shepherd.js version pin (latest stable 14.x at planning time).
- **CD-02:** Visual style of the "Show tour" floating button (icon, color, position offset) — match existing tailwind utility patterns.
- **CD-03:** Whether to move `tests/fixtures/scenario_runner/schema.py` to `app/services/scenario_schema.py` (or similar) so both the test runner and the new `/api/tours/` router can import it cleanly, vs creating a thin import shim. **Planner decides** based on import-graph cleanliness.
- **CD-04:** `sessionStorage` key naming and resume-after-navigation state shape for `open` op continuation.
- **CD-05:** Whether the first shipped tour lives at `scenarios/tours/keyword-suggest-walkthrough.yaml` (subdir) or `scenarios/03-keyword-suggest-walkthrough.yaml` (flat). Either is fine — pick whichever feels cleaner after seeing the file count.

### Folded Todos
None — no pending todos matched this phase.

</decisions>

<specifics>
## Specific Ideas

- Tour player and 19.1 test runner share the YAML schema as a **locked contract** — any breaking change to schema requires updating both runners simultaneously (already noted in `scenarios/README.md`).
- The first shipped tour should walk through one of the v2.1 onboarding flows (Phase 18 Project Health checklist or Phase 15 Keyword Suggest) — exact pick is planner's call based on which flow is most useful for new admins to learn.
- `wait_for_click` is the killer feature for onboarding tours: it forces the user to perform the actual action (e.g. "click 'Add Site'") instead of just reading. Make sure the Shepherd integration supports this — Shepherd has `advanceOn: { selector, event }` which maps directly.

</specifics>

<deferred>
## Deferred Ideas

- **Migrating 19.1 test scenarios into tours** — would require auto-fill UX which we explicitly rejected. Skip permanently unless requirements change.
- **Multi-language tour text** — single `intro`/`say` string only; no i18n keys. Add later if the platform gets a second language.
- **Tour analytics / completion tracking** — no DB writes from the player. If we want "X% of admins completed the tour" later, that's a new phase with a new table.
- **Per-user "tour seen" state** — admins can replay any tour any time. No "don't show again" checkbox in v19.2.
- **Non-admin user tours** — platform is admin-only today; revisit if/when client-facing roles arrive.
- **Tour authoring UI** — tours are authored as YAML files in git, same as test scenarios. No in-app editor.
- **Multi-tour picker UI on a page** — if only one tour ships in 19.2, planner may defer the picker and just auto-launch the single match.

</deferred>
