## 1. Tour Engine

| Option | Description | Selected |
|--------|-------------|----------|
| Shepherd.js via CDN | ~30KB, themed, popper positioning, modal overlay, prev/next, keyboard nav out of the box. Fits existing CDN pattern in base.html (htmx/tailwind/flatpickr). | ✓ |
| Custom `tour.js` | Zero deps, ~200 LOC, total styling control. Would re-implement Shepherd features without community bugfixes. | |

**User's choice:** All recommendations ("лк").
**Notes:** Shepherd's `advanceOn: { selector, event }` maps directly to the reserved `wait_for_click` op — clean fit.

---

## 2. Step Op Handling Inside a Scenario

| Option | Description | Selected |
|--------|-------------|----------|
| (a) Skip all test-only ops | Tour player ignores `open`/`click`/`fill`/`wait_for`/`expect_*` with a warning, mirroring how 19.1 skips reserved tour ops. | |
| (b) Execute test-only ops | Tour player drives the user through the flow by auto-clicking and auto-filling. | |
| (c) Hybrid — execute `open`, skip the rest | `open` → navigate (tour can start from anywhere); `click`/`fill`/`expect_*` → console.warn and skip. | ✓ |

**User's choice:** All recommendations.
**Notes:** Auto-filling forms or auto-clicking would be confusing UX — the user wants to learn, not watch a robot. Auto-`open` is necessary so a tour can launch from any page and route the user to the entry URL.

## 3. Tour↔Page Binding

| Option | Description | Selected |
|--------|-------------|----------|
| (a) Filename + URL prefix convention | E.g. scenario `name` field declares `pages: [/ui/keyword-suggest/]`. | |
| (b) New optional top-level `tour:` block in YAML | `entry_url`, `title`, `intro` — backward-compatible Pydantic addition. | ✓ |
| (c) Separate `scenarios/tours/` subdirectory | Tour-only files live in their own folder. | |
| (d) Per-page admin config in DB | Map page URL → scenario name in a new table. | |

**User's choice:** All recommendations.
**Notes:** One file, two contracts. No new directories, no DB table, no filename magic. Schema discriminated union stays untouched — only a new optional top-level field.

## 4. Authoring Model

| Option | Description | Selected |
|--------|-------------|----------|
| Reuse 19.1 test scenarios as-is | The two existing scenarios fill forms with "smoke seed" — confusing as user-facing tours. | |
| Tour-only YAML files (same `scenarios/` dir, same schema) | Authored separately, use only tour-native ops + optional `open`. Test scenarios stay test-only. | ✓ |

**User's choice:** All recommendations.
**Notes:** Phase 19.2 ships at least one real tour as a phase-gate; planner picks the most valuable first onboarding flow from v2.1.

---

## 5. YAML → Browser Pipeline

| Option | Description | Selected |
|--------|-------------|----------|
| (a) FastAPI endpoint serving validated JSON | `GET /api/tours/?page=...` (list) + `GET /api/tours/{name}` (full tour). Server reuses 19.1 Pydantic schema. | ✓ |
| (b) Serve raw YAML + parse with `js-yaml` CDN | Adds ~15KB to the browser, no server-side validation. | |
| (c) Inline-render JSON into the page | Server injects the tour JSON into the HTML when admin clicks "Show tour". | |

**User's choice:** All recommendations.
**Notes:** Server already has Pydantic + PyYAML — reuse, don't re-implement in JS. Browser stays lean. Endpoint can filter by `entry_url` prefix against `location.pathname` so the frontend gets only relevant tours in one round-trip.

## 6. "Show Tour" Button Placement & Gating

| Option | Description | Selected |
|--------|-------------|----------|
| (a) Floating button bottom-right on every page, hidden for non-admin | Always rendered for admins, regardless of whether a tour exists for the page. | |
| (b) Item in main nav/header, admin-only | Permanent nav slot. | |
| (c) Only on pages with a registered tour, admin-only | `fetch('/api/tours/?page=...')`; render floating button only if response is non-empty AND user is admin. | ✓ |

**User's choice:** All recommendations.
**Notes:** Zero per-template work, no UI noise on pages without tours, single auth gate via the existing dependency.
