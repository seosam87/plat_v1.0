## Summary

Phase 999.9 adds an AI agent catalogue — a library of named prompt templates with `{{variable}}` placeholders, a form-based launcher, and Celery-backed LLM execution. The pattern is a clean extension of the existing LLM briefs flow (`app/routers/llm_briefs.py` + `app/tasks/llm_tasks.py`), reusing the same Anthropic SDK wrapper, circuit breaker, and usage logging infrastructure.

The central design insight is that this phase introduces **two new first-class entities**: `AIAgent` (the template, with system prompt + user_template + LLM config) and `AgentJob` (one execution run). `AIAgent` is analogous to `ContentBrief` (a static definition) and `AgentJob` is analogous to `LLMBriefJob` (a job that processes it). The UI follows the Playbook Builder card-grid + HTMX filter pattern, already established in `app/routers/playbooks.py`.

The `{{variable}}` parsing is a pure-Python regex pass over `user_template` — no external library needed. Variable names are extracted, deduplicated, and used to auto-generate form fields on the run page. The only significant new UI component is the variable-preview HTMX endpoint (`/ui/agents/parse-vars`) that re-renders detected variables live as the user types.

**Primary recommendation:** Mirror the LLMBriefJob pattern exactly for the job model and Celery task. Use the BlockCategory/PlaybookBlock structural pattern for AgentCategory seeding. Add a single `agents` nav section in `app/navigation.py`.

### Pattern 2: Variable Parser

**What:** Pure Python regex extraction of `{{variable_name}}` tokens from `user_template`.

```python
# Source: Claude's Discretion — no external library needed
import re

_VAR_RE = re.compile(r"\{\{(\w+)\}\}")

def parse_template_variables(template: str) -> list[str]:
    """Extract unique ordered variable names from a {{var}} template.
    Returns list preserving first-seen order, deduplicated.
    """
    seen = set()
    result = []
    for match in _VAR_RE.finditer(template):
        name = match.group(1)
        if name not in seen:
            seen.add(name)
            result.append(name)
    return result

def render_template(template: str, inputs: dict[str, str]) -> str:
    """Substitute {{var}} placeholders with provided input values."""
    def replacer(m):
        return inputs.get(m.group(1), m.group(0))
    return _VAR_RE.sub(replacer, template)
```

**When to use:** In `agent_service.py` for both the parse-vars endpoint and the Celery task when building the final user message.

### Pattern 3: Celery Task (mirrors app/tasks/llm_tasks.py)

**What:** `run_agent_job` Celery task following the exact same structure as `generate_llm_brief_enhancement`.

```python
# Source: app/tasks/llm_tasks.py — mirror exactly

@celery_app.task(
    name="agents.run_agent_job",
    bind=True,
    max_retries=3,
    queue="default",
)
def run_agent_job(self, job_id: int) -> None:
    loop = asyncio.new_event_loop()
    try:
        loop.run_until_complete(_run_agent_job(job_id, self.request.retries, self.max_retries))
    except _TransientError as exc:
        countdown = 60 * 2 ** self.request.retries
        raise self.retry(exc=exc, countdown=countdown)
    finally:
        loop.close()
```

Key differences from `generate_llm_brief_enhancement`:
- Takes `inputs_json` from `AgentJob` (variable values submitted by user)
- Uses `agent.system_prompt` directly as the `system=` parameter in `messages.create()`
- Renders `agent.user_template` using `render_template(agent.user_template, inputs)`
- Respects `agent.model`, `agent.temperature`, `agent.max_tokens` instead of global config constants
- Does NOT use `json_schema` output format unless `agent.output_format == "json"` (in which case parse JSON; text/markdown stored raw)
- Increments `AIAgent.usage_count` on success

### Pattern 4: Anthropic API call with per-agent settings

**What:** The existing `call_llm_brief_enhance` is hardcoded to `ANTHROPIC_MODEL` and the brief schema. For agents, the call needs to be flexible.

```python
# Source: app/services/llm/llm_service.py — adapted

async def call_agent(
    api_key: str,
    *,
    system_prompt: str,
    user_message: str,
    model: str,
    temperature: float,
    max_tokens: int,
) -> tuple[str, int, int]:
    """Call Anthropic for agent execution. Returns (output_text, in_tok, out_tok)."""
    from anthropic import AsyncAnthropic
    client = AsyncAnthropic(api_key=api_key)
    response = await client.messages.create(
        model=model,
        max_tokens=max_tokens,
        system=system_prompt,
        messages=[{"role": "user", "content": user_message}],
        # temperature is a valid param in messages.create
        temperature=temperature,
    )
    text = response.content[0].text
    return text, response.usage.input_tokens, response.usage.output_tokens
```

Add `call_agent()` to `app/services/llm/llm_service.py` (or a new `app/services/agent_llm_service.py`). Prefer adding to `llm_service.py` to co-locate Anthropic SDK usage.

### Anti-Patterns to Avoid

- **Do NOT reuse ExpertSource or BlockCategory models.** D-02 explicitly states this is a separate entity with its own catalogue. Sharing tables would couple unrelated domains.
- **Do NOT call `messages.create()` synchronously from the Celery sync wrapper.** Always wrap in `asyncio.new_event_loop()` + `loop.run_until_complete()` as `llm_tasks.py` does.
- **Do NOT store variable values in the Anthropic call directly.** Render `user_template` client-side in the task using `render_template()` before sending to the API — never pass raw template with `{{}}` tokens.
- **Do NOT use a JSON schema constraint for text/markdown output formats.** The JSON schema output_config only applies when `output_format == "json"`. For text and markdown, call plain `messages.create()` without `output_config`.
- **Do NOT poll on page load if job_id is None.** The run page should only render the polling container after a POST creates a job.
- **Do NOT increment `usage_count` before confirming the job succeeded.** Increment only in the `status="done"` success path of the Celery task.

## Common Pitfalls

### Pitfall 1: HTMX polling div loses its ID on swap
**What goes wrong:** If the polling partial doesn't include `id="agent-run-result"` in its outerHTML, the `hx-swap="outerHTML"` replaces the div with content that has no ID — next polling cycle targets nothing.
**Why it happens:** Each rendered state (pending/done/error) must carry `id="agent-run-result"` even in the "done" state (it just lacks `hx-trigger`).
**How to avoid:** The `_agent_job_status.html` partial always wraps content in `<div id="agent-run-result" ...>`.
**Warning signs:** Network tab shows 200 responses to the poll URL but UI never updates.

### Pitfall 2: Variable parser strips valid variable names
**What goes wrong:** Regex `\w+` matches `[a-zA-Z0-9_]` — but users may attempt `{{my-var}}` with hyphens, which `\w+` silently ignores.
**How to avoid:** Document in form hint that variable names must be `[A-Za-z0-9_]` only. Show parsed variable preview immediately so users see missing variables before saving.

### Pitfall 3: `temperature` field precision in SQLAlchemy
**What goes wrong:** Using `Float` for temperature can produce floating-point repr noise (0.7000000001).
**How to avoid:** Use `Numeric(3, 2)` at the column level; coerce to float only when passing to the Anthropic SDK.

### Pitfall 4: AgentJob `agent_id` ondelete="SET NULL" — deleted agent orphans jobs
**What goes wrong:** Deleting an AIAgent nullifies `AgentJob.agent_id` FK — existing job result pages will get a 404 when trying to load the agent name.
**How to avoid:** Store `agent_name` as a denormalized string on `AgentJob` at job creation time (snapshot), so result page renders correctly even after agent deletion.

### Pitfall 5: Celery task not in `celery_app.include`
**What goes wrong:** New `app/tasks/agent_tasks.py` task is registered but `celery_app.py` `include=` list is not updated — workers never see the task; `.delay()` queues it but nothing processes it.
**How to avoid:** Add `"app.tasks.agent_tasks"` to the `include=` list in `app/celery_app.py`.

### Pitfall 6: Router not registered in `app/main.py`
**What goes wrong:** FastAPI silently ignores routes from unregistered routers — all `/ui/agents/*` endpoints return 404.
**How to avoid:** Add `from app.routers.agents import router as agents_router` + `app.include_router(agents_router)` to `app/main.py`.

### Pitfall 7: Migration revision chain
**What goes wrong:** New migration must chain from the current HEAD. Looking at the versions directory: `0055`, `0056`, `9c65e7d94183_add_report_schedules_table.py` and `0056_add_tg_notifications_toggle.py` may be a branched head.
**How to avoid:** Before writing the migration, run `alembic heads` to detect any existing multiple heads. If there are multiple heads, the migration must use `down_revision` that references the correct merged head. The playbook migration (`0054`) is at revision `0054`. Check the chain from 0054 → 0055 → 0056 → check for the report_schedules head separately.

### Variable Parse-Vars Endpoint (HTMX GET)

```python
# Source: Claude's Discretion — adapted from playbooks filter pattern

@router.get("/ui/agents/parse-vars", response_class=HTMLResponse)
async def parse_vars(
    request: Request,
    template: str = "",
    user: User = Depends(require_any_authenticated),
) -> HTMLResponse:
    """HTMX partial: parse {{vars}} from template string, return badge list."""
    from app.services.agent_service import parse_template_variables
    variables = parse_template_variables(template)
    if not variables:
        return HTMLResponse(
            '<p class="text-xs text-gray-400 italic">'
            'Переменных не обнаружено. Используйте <code>{{переменная}}</code> в шаблоне запроса.'
            '</p>'
        )
    badges = "".join(
        f'<span class="inline-block px-2 py-0.5 text-xs rounded bg-slate-100 text-slate-600 mr-1 mb-1">'
        f'{{{{{v}}}}}'
        f'</span>'
        for v in variables
    )
    return HTMLResponse(f'<div class="flex flex-wrap">{badges}</div>')
```
