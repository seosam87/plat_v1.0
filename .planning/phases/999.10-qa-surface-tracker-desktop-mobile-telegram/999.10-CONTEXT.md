# Phase 999.10: QA Surface Tracker — Desktop, Mobile, Telegram - Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Retroactive QA audit system. A registry of user-facing features (FeatureSurface) tracked across three surfaces (desktop, mobile, Telegram) with periodic retest policies. Enables the team to see which features are verified, which are overdue for retesting, and where coverage gaps exist.

</domain>

<decisions>
## Implementation Decisions

### Granularity
- **D-01:** A "feature" is a user flow / scenario (e.g., "Run position check and see results"), NOT individual routes or screens. Expected ~15-20 entries in the registry.

### Surface Matrix
- **D-02:** Parent-child model: one FeatureSurface record per user flow + child SurfaceCheck records per surface (desktop, mobile, telegram). Each SurfaceCheck tracks its own status, last_tested date, and notes independently.

### Retest Policy & Triggers
- **D-03:** Retest is triggered by periodicity — each FeatureSurface has a retest_policy (interval). Celery Beat periodic task scans for overdue items and marks them "needs retest".
- **D-04:** No git-diff trigger. No manual-only trigger. Pure time-based with Celery Beat.

### Registry Population
- **D-05:** Auto-discovery + manual curation. `discover_routes()` (from Phase 15.1 smoke infrastructure) proposes route candidates. Admin groups routes into user-flow scenarios and approves them through UI.

### Claude's Discretion
- DB schema design (column types, indexes, constraints)
- UI layout for the registry and matrix view
- Celery Beat task implementation details
- How `discover_routes()` candidates are presented for grouping

</decisions>
