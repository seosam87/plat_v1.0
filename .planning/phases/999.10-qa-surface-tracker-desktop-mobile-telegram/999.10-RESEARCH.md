## Summary

Phase 999.10 builds a lightweight retroactive QA audit registry. It consists of two SQLAlchemy models (`FeatureSurface` + `SurfaceCheck`), a FastAPI/Jinja2/HTMX admin UI under `/ui/qa/`, and a Celery Beat task that scans for overdue surfaces and marks their checks as `needs_retest`. There are no external dependencies — everything uses already-installed stack components.

The parent-child model is the only sensible design: one row per user-flow captures metadata (name, description, retest interval), and three child rows per flow capture per-surface QA state independently. The matrix view (flows as rows, surfaces as columns) is a natural Jinja2 table render.

Auto-discovery surfaces all GET routes via the existing `discover_routes(app)` helper from `tests/_smoke_helpers.py`. An admin-only candidate review page lets the operator group routes into named flows and save them as `FeatureSurface` records. This is entirely UI-driven — no code generation.

**Primary recommendation:** Implement in three waves: (1) DB models + Alembic migration, (2) router + templates (registry list, matrix view, CRUD, candidate discovery page), (3) Celery Beat `scan_overdue_surfaces` task + redbeat registration.

### Pattern 1: Parent-Child SQLAlchemy Models

Two tables. `feature_surfaces` is the parent (one row per user flow). `surface_checks` is the child with a `surface` enum (`desktop | mobile | telegram`) — one child row per (feature, surface) pair.

```python
# app/models/qa_surface.py
import uuid
import enum
from datetime import datetime
import sqlalchemy as sa
from sqlalchemy.dialects.postgresql import UUID, INTERVAL
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.database import Base


class Surface(str, enum.Enum):
    desktop  = "desktop"
    mobile   = "mobile"
    telegram = "telegram"


class CheckStatus(str, enum.Enum):
    not_tested   = "not_tested"
    passed       = "passed"
    failed       = "failed"
    needs_retest = "needs_retest"


class FeatureSurface(Base):
    __tablename__ = "feature_surfaces"

    id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), primary_key=True, default=uuid.uuid4
    )
    slug: Mapped[str] = mapped_column(sa.String(80), unique=True, nullable=False)
    name: Mapped[str] = mapped_column(sa.String(200), nullable=False)
    description: Mapped[str | None] = mapped_column(sa.Text, nullable=True)
    # retest_policy stored as days integer — simple to display and compare
    retest_days: Mapped[int] = mapped_column(sa.Integer, nullable=False, default=30)
    is_active: Mapped[bool] = mapped_column(
        sa.Boolean, nullable=False, server_default=sa.text("true")
    )
    created_at: Mapped[datetime] = mapped_column(
        sa.DateTime(timezone=True), nullable=False, server_default=sa.func.now()
    )
    updated_at: Mapped[datetime] = mapped_column(
        sa.DateTime(timezone=True), nullable=False,
        server_default=sa.func.now(), onupdate=datetime.utcnow
    )

    checks: Mapped[list["SurfaceCheck"]] = relationship(
        "SurfaceCheck", back_populates="feature", cascade="all, delete-orphan"
    )


class SurfaceCheck(Base):
    __tablename__ = "surface_checks"
    __table_args__ = (
        sa.UniqueConstraint("feature_id", "surface", name="uq_feature_surface"),
    )

    id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), primary_key=True, default=uuid.uuid4
    )
    feature_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True),
        sa.ForeignKey("feature_surfaces.id", ondelete="CASCADE"),
        nullable=False,
        index=True,
    )
    surface: Mapped[Surface] = mapped_column(
        sa.Enum(Surface), nullable=False
    )
    status: Mapped[CheckStatus] = mapped_column(
        sa.Enum(CheckStatus), nullable=False, default=CheckStatus.not_tested
    )
    last_tested_at: Mapped[datetime | None] = mapped_column(
        sa.DateTime(timezone=True), nullable=True
    )
    notes: Mapped[str | None] = mapped_column(sa.Text, nullable=True)
    tested_by: Mapped[str | None] = mapped_column(sa.String(100), nullable=True)
    updated_at: Mapped[datetime] = mapped_column(
        sa.DateTime(timezone=True), nullable=False,
        server_default=sa.func.now(), onupdate=datetime.utcnow
    )

    feature: Mapped[FeatureSurface] = relationship(
        "FeatureSurface", back_populates="checks"
    )
```

**Key design choices (Claude's discretion applied):**
- `retest_days: int` rather than PostgreSQL `INTERVAL` — simpler to display/edit in HTML forms and trivial to compare with `timedelta(days=retest_days)` in Python.
- `surface` stored as SQLAlchemy `Enum` → PostgreSQL enum type — type-safe, efficient storage.
- `UniqueConstraint("feature_id", "surface")` ensures exactly one check row per (feature, surface) combination.
- `slug` on `FeatureSurface` for URL-safe identification (e.g., `/ui/qa/position-check`).
- No FK to `sites` — the registry is platform-wide, not per-site (surfaces exist independently of which site is selected).

### Pattern 2: Service Layer Auto-Creates Child Rows

When a `FeatureSurface` is created, the service automatically creates all three `SurfaceCheck` rows. This guarantees the matrix view always has a complete 3-column row.

```python
# app/services/qa_surface_service.py
async def create_feature_surface(
    db: AsyncSession,
    slug: str,
    name: str,
    description: str | None,
    retest_days: int,
) -> FeatureSurface:
    feature = FeatureSurface(
        slug=slug, name=name, description=description, retest_days=retest_days
    )
    db.add(feature)
    await db.flush()   # get feature.id
    for surface in Surface:
        db.add(SurfaceCheck(
            feature_id=feature.id,
            surface=surface,
            status=CheckStatus.not_tested,
        ))
    await db.commit()
    await db.refresh(feature)
    return feature
```

### Pattern 3: Matrix View Query

Eager-load checks with the parent in a single query to avoid N+1:

```python
from sqlalchemy import select
from sqlalchemy.orm import selectinload

async def list_features_with_checks(db: AsyncSession) -> list[FeatureSurface]:
    result = await db.execute(
        select(FeatureSurface)
        .options(selectinload(FeatureSurface.checks))
        .order_by(FeatureSurface.name)

    return result.scalars().all()
```

### Pattern 4: Celery Beat Scan Task (redbeat, proven pattern)

Follows exactly the same registration pattern as `register_notifications_cleanup_schedule()` in `app/tasks/notification_tasks.py`:

```python
# app/tasks/qa_surface_tasks.py
from app.celery_app import celery_app

@celery_app.task(
    name="app.tasks.qa_surface_tasks.scan_overdue_surfaces",
    bind=True,
    max_retries=3,
    queue="default",
    soft_time_limit=120,
    time_limit=150,
)
def scan_overdue_surfaces(self) -> dict:
    """Mark SurfaceCheck rows overdue based on retest_days policy."""
    loop = asyncio.new_event_loop()
    try:
        return loop.run_until_complete(_scan())
    except Exception as exc:
        raise self.retry(exc=exc, countdown=60)
    finally:
        loop.close()


async def _scan() -> dict:
    from datetime import timezone, timedelta
    from sqlalchemy import select, update
    from app.database import AsyncSessionLocal
    from app.models.qa_surface import FeatureSurface, SurfaceCheck, CheckStatus

    now = datetime.now(timezone.utc)
    async with AsyncSessionLocal() as db:
        features = await db.execute(
            select(FeatureSurface)
            .options(selectinload(FeatureSurface.checks))
            .where(FeatureSurface.is_active.is_(True))
        )
        marked = 0
        for feature in features.scalars().all():
            deadline = now - timedelta(days=feature.retest_days)
            for check in feature.checks:
                if check.status == CheckStatus.passed:
                    if check.last_tested_at is None or check.last_tested_at < deadline:
                        check.status = CheckStatus.needs_retest
                        marked += 1
        await db.commit()
    return {"status": "ok", "marked": marked}


def register_qa_surface_scan_schedule() -> None:
    """Register daily overdue scan in RedBeat (idempotent). Called from beat_init."""
    from celery.schedules import crontab
    from redbeat import RedBeatSchedulerEntry

    key = "qa-surface-scan-daily"
    try:
        RedBeatSchedulerEntry.from_key(f"redbeat:{key}", app=celery_app).delete()
    except KeyError:
        pass
    entry = RedBeatSchedulerEntry(
        name=key,
        task="app.tasks.qa_surface_tasks.scan_overdue_surfaces",
        schedule=crontab(hour=4, minute=0),
        app=celery_app,
    )
    entry.save()
```

### Anti-Patterns to Avoid

- **Calling `discover_routes()` at module import time in router** — it imports `app.main` which would create circular imports. Call it inside the request handler.
- **Storing `retest_policy` as PostgreSQL `INTERVAL`** — requires special handling in Python and in HTML form inputs. Use `retest_days: int` instead.
- **Many-to-many surface table** — over-engineered for exactly 3 surfaces. Fixed `surface` enum is simpler and the UniqueConstraint enforces the invariant.
- **Scanning in a FastAPI background task** — must be Celery Beat, not `BackgroundTasks`, per D-03/D-04.
- **Not adding `/ui/qa` to `UI_PREFIXES`** — the smoke crawler will silently miss the new routes. Must be added at the same time as the router.
- **Not adding `/m/` to `UI_PREFIXES`** — the CONTEXT.md notes `/m/` is currently missing from smoke coverage. The candidates page will surface mobile routes, but they won't be smoke-tested unless `/m/` is added to `UI_PREFIXES`.

## Common Pitfalls

### Pitfall 1: Circular Import via `app.main` in Router

**What goes wrong:** `qa_surfaces.py` imports from `tests/_smoke_helpers.py` which calls `discover_routes(app)`. If `from app.main import app` is at module level, it causes a circular import at FastAPI startup.

**Why it happens:** `app.main` imports all routers including `qa_surfaces`, so importing `app.main` inside `qa_surfaces` at module level creates a cycle.

**How to avoid:** Import `app` only inside the function body of the candidates endpoint handler — never at module level.

**Warning signs:** `ImportError: cannot import name 'app' from partially initialized module 'app.main'` on startup.

### Pitfall 2: Missing `/m/` in `UI_PREFIXES` for Telegram/Mobile Coverage

**What goes wrong:** Mobile routes under `/m/` are not included in `UI_PREFIXES`, so `discover_routes()` never returns them. The candidate page shows 0 mobile routes. The QA surface for "mobile" surface always has no route evidence.

**Why it happens:** `_smoke_helpers.py` `UI_PREFIXES` was deliberately built without `/m/` (see comment in CONTEXT.md: "NOT in UI_PREFIXES (no smoke coverage)").

**How to avoid:** Add `"/m/"` to `UI_PREFIXES` as part of this phase. This also gives mobile routes their first smoke coverage.

**Warning signs:** Candidates page only shows desktop routes; mobile surface checks cannot be auto-populated with discovery evidence.

### Pitfall 3: `scan_overdue_surfaces` Marks `failed` Checks as `needs_retest`

**What goes wrong:** The scan task changes `failed` checks to `needs_retest`, obscuring the fact that a check was actively failing (not just untested).

**Why it happens:** Naive `if check.status != not_tested` logic.

**How to avoid:** The scan should only transition `passed` → `needs_retest`. `failed` checks remain `failed` until manually resolved. `not_tested` checks remain `not_tested` (the interval policy only applies once a baseline `passed` state exists).

**Warning signs:** QA matrix shows `needs_retest` everywhere even on features that were recently marked `failed`.

### Pitfall 4: `onupdate=datetime.utcnow` Not Timezone-Aware

**What goes wrong:** SQLAlchemy `onupdate=datetime.utcnow` (without `timezone.utc`) writes naive datetimes to a `DateTime(timezone=True)` column. PostgreSQL stores them but comparison with timezone-aware `datetime.now(timezone.utc)` in the scan task produces unexpected ordering.

**Why it happens:** `datetime.utcnow()` returns a naive datetime; `DateTime(timezone=True)` is a `TIMESTAMPTZ` column.

**How to avoid:** Use `onupdate=lambda: datetime.now(timezone.utc)` throughout. (Note: existing models like `project.py` use `default=datetime.utcnow` — this is a known existing inconsistency, but new models should use timezone-aware defaults.)

**Warning signs:** Overdue scan marks freshly-tested items as `needs_retest` immediately after testing.

### Pitfall 5: Smoke Seed Must Include `feature_surface_id` and `surface_check_id`

**What goes wrong:** The smoke crawler hits `/ui/qa/{feature_id}` and `/ui/qa/checks/{check_id}` but `SMOKE_IDS` has no `feature_surface_id` or `surface_check_id` entries.

**Why it happens:** `resolve_path()` raises `UnknownParamError` for any path param not in `PARAM_MAP`.

**How to avoid:** Add `feature_surface_id` and `surface_check_id` to `SMOKE_IDS` in `tests/fixtures/smoke_seed.py` and seed the corresponding rows in `seed_extended()`.

**Warning signs:** `UnknownParamError: Unknown path param 'feature_surface_id'` in smoke test output.

## Open Questions

1. **Should the candidates page allow drag-and-drop grouping of routes into flows?**
   - What we know: HTMX 2.0 supports `sortable` via the official extension; the platform uses HTMX 2.0.
   - What's unclear: Whether drag-and-drop is worth the complexity for ~73 routes. A simple checkbox + name form is likely sufficient.
   - Recommendation: Use checkboxes + text input for the MVP. Drag-and-drop is deferred.

2. **Should `SurfaceCheck.tested_by` be a FK to `users.id` or a free-text username?**
   - What we know: Notifications and audit_log store `user_id` FKs. But QA testing could be done by external testers without platform accounts.
   - What's unclear: Whether non-user testers are realistic for this team.
   - Recommendation: Free-text `String(100)` for simplicity. The user's username is auto-populated from the current session (`user.username`) but can be overridden.

3. **`/m/` routes in `UI_PREFIXES` — will adding it break the existing smoke baseline count?**
   - What we know: `test_ui_smoke.py` comment documents current baseline as 73 routes. Adding `/m/` will increase it.
   - What's unclear: Exact number of HTML GET routes in `app/routers/mobile.py` (it has ~38 routes but not all are HTML GET).
   - Recommendation: Update the baseline comment in `test_ui_smoke.py` when `/m/` is added, after running the suite to see the new count. Add mobile route params to `SMOKE_IDS` and `SMOKE_SKIP` as needed.
