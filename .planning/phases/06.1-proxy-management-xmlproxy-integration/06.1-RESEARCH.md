## Summary

Phase 06.1 replaces the existing Playwright-based `proxy_serp_service.py` with XMLProxy.ru for all Yandex SERP parsing. The XMLProxy API is wire-compatible with the old Yandex XML API: it takes the same GET parameters (`user`, `key`, `query`, `lr`, `groupby`, etc.) and returns the same XML format with `<group><doc><url>` structure. The main additions are: a -55 error code (async/rate limit — retry after 5 minutes), a balance API endpoint, and a `ydomain` parameter for region selection.

The project already has the full infrastructure this phase needs: Fernet encryption in `crypto_service.py`, Celery retry patterns throughout, Telegram alerts via `telegram_service.py`, HTMX-driven admin CRUD, and SQLAlchemy 2.0 async models. This phase follows every existing pattern — it is an additive phase with no architectural changes.

The only new library dependency is `httpx[socks]` (adds `socksio` for SOCKS5 proxy health checks). All other needs are met by already-installed packages: `httpx` for XMLProxy HTTP calls, stdlib `xml.etree.ElementTree` for XML parsing, `cryptography` for Fernet encryption of credentials.

**Primary recommendation:** Build `xmlproxy_service.py` as a standalone sync service using `httpx.Client` (sync, not async — consistent with Celery task context), parse XML responses with stdlib `xml.etree.ElementTree`, store XMLProxy/anticaptcha credentials in a single `service_credentials` table, and store proxies in a `proxies` table. The position pipeline gets a new `_check_via_xmlproxy()` helper that mirrors `_check_via_dataforseo()` in style.

## XMLProxy.ru API Reference

Source: CONTEXT.md canonical refs (provided by user who has active account).

### Search Endpoint

```
GET https://xmlproxy.ru/search.php
  ?user=<login>
  &key=<api_key>
  &query=<url_encoded_query>
  &lr=<yandex_region_id>
  &groupby=attr=d.mode=deep.groups-on-page=10
  &ydomain=ru
```

Common Yandex region IDs: 213 (Moscow), 2 (Saint Petersburg), 11316 (Novosibirsk), 54 (Yekaterinburg).

### Balance Endpoint

```
GET https://xmlproxy.ru/balance.php?user=<login>&key=<key>
```

Returns JSON: `{"data": 1234.56, "cur_cost": 0.018, "max_cost": 0.05}`
- `data`: current balance in rubles
- `cur_cost`: cost per query at current tariff
- `max_cost`: maximum allowed cost per query

### Error Codes

| Code | Meaning | Action |
|------|---------|--------|
| -55 | Async request queued | `self.retry(countdown=300, max_retries=3)` |
| -32 | Insufficient funds | Pause Yandex parsing + Telegram alert |
| -34 | Invalid auth credentials | Log error + Telegram alert to admin |
| -132 | Rate limit exceeded | Retry with backoff |

Errors appear as `<error code="-55">message</error>` in the XML response body.

## DB Model Design (Claude's Discretion)

### ServiceCredential Model

Single table with `service_name` as unique key. Stores all third-party API credentials. Sensitive fields Fernet-encrypted before storage using existing `crypto_service.py`.

```python
# app/models/service_credential.py
import uuid
from datetime import datetime
from sqlalchemy import DateTime, String, Text
from sqlalchemy.dialects.postgresql import UUID
from sqlalchemy.orm import Mapped, mapped_column
from app.database import Base

class ServiceCredential(Base):
    __tablename__ = "service_credentials"

    id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    service_name: Mapped[str] = mapped_column(String(100), nullable=False, unique=True)
    # Values: "xmlproxy", "rucaptcha"
    credential_data: Mapped[str | None] = mapped_column(Text, nullable=True)
    # JSON text: {"user": "login", "key": "<fernet_encrypted_key>"}
    # Only the key/password fields are encrypted; username is plaintext
    created_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), default=datetime.utcnow, nullable=False)
    updated_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), default=datetime.utcnow, onupdate=datetime.utcnow, nullable=False)
```

Usage pattern matches how WP passwords are handled in `site_service.py`:
- On write: `json.dumps({"user": user, "key": encrypt(api_key)})`
- On read: parse JSON, `decrypt(data["key"])`

## Architecture Patterns

### Pattern: XMLProxy Service (sync — for Celery context)

Use `httpx.Client` (sync) not `httpx.AsyncClient`. Celery tasks in this project use `get_sync_db()` and sync DB access — async adds event loop ceremony. The `send_message_sync()` pattern in `telegram_service.py` confirms this approach.

```python
# app/services/xmlproxy_service.py
import xml.etree.ElementTree as ET
import httpx
from loguru import logger

XMLPROXY_SEARCH = "https://xmlproxy.ru/search.php"
XMLPROXY_BALANCE = "https://xmlproxy.ru/balance.php"


class XMLProxyError(Exception):
    def __init__(self, code: int, message: str):
        self.code = code
        self.message = message
        super().__init__(f"XMLProxy error {code}: {message}")


def search_yandex_sync(user: str, key: str, query: str, lr: int = 213) -> dict:
    """Query XMLProxy for Yandex organic results (sync, for Celery tasks).

    Returns: {"results": [...], "error_code": int | None}
    Raises: XMLProxyError on -55, -32, -34, -132
    """
    params = {
        "user": user, "key": key, "query": query,
        "lr": str(lr), "groupby": "attr=d.mode=deep.groups-on-page=10",
        "ydomain": "ru",
    }
    with httpx.Client(timeout=30) as client:
        resp = client.get(XMLPROXY_SEARCH, params=params)
        resp.raise_for_status()
    return _parse_yandex_xml(resp.text)


def fetch_balance_sync(user: str, key: str) -> dict | None:
    """Fetch XMLProxy balance. Returns {"data": float, "cur_cost": float, "max_cost": float}."""
    params = {"user": user, "key": key}
    try:
        with httpx.Client(timeout=10) as client:
            resp = client.get(XMLPROXY_BALANCE, params=params)
            resp.raise_for_status()
            return resp.json()
    except Exception as exc:
        logger.warning("XMLProxy balance fetch failed", error=str(exc))
        return None


def _parse_yandex_xml(xml_text: str) -> dict:
    root = ET.fromstring(xml_text)
    error_el = root.find(".//error")
    if error_el is not None:
        code = int(error_el.get("code", "0"))
        raise XMLProxyError(code, error_el.text or "")
    results = []
    for i, group in enumerate(root.findall(".//group"), 1):
        doc = group.find("doc")
        if doc is None:
            continue
        url = doc.findtext("url", "")
        title = doc.findtext("title", "")
        domain = doc.findtext("domain", "")
        if not domain and url:
            parts = url.split("/")
            domain = parts[2] if len(parts) > 2 else ""
        results.append({"position": i, "url": url, "title": title, "domain": domain})
    return {"results": results, "error_code": None}
```

### Pattern: Proxy Health Check

```python
# app/services/proxy_health_service.py
# Source: httpx official docs — SOCKS5 needs socksio (httpx[socks])
import time
import httpx
from loguru import logger

HEALTH_CHECK_URL = "https://ya.ru/robots.txt"


def check_proxy_sync(proxy_url: str) -> tuple[str, int | None]:
    """Test proxy. Returns (status, response_time_ms). Status: 'active' | 'dead'."""
    start = time.monotonic()
    try:
        with httpx.Client(proxy=proxy_url, timeout=10) as client:
            resp = client.get(HEALTH_CHECK_URL)
            resp.raise_for_status()
        ms = int((time.monotonic() - start) * 1000)
        return "active", ms
    except ImportError:
        # socksio not installed — SOCKS5 unsupported
        logger.error("SOCKS5 proxy health check failed: install httpx[socks]")
        return "dead", None
    except Exception as exc:
        logger.warning("Proxy health check failed", proxy=proxy_url, error=str(exc))
        return "dead", None
```

## Common Pitfalls

### Pitfall 1: -55 Is Not a Permanent Error

**What goes wrong:** Treating -55 as permanent failure, writing `position=None` to DB, burning all 3 retries in 5 minutes.
**Why it happens:** -55 means "request is queued asynchronously." A retry within 12h is free on XMLProxy's side.
**How to avoid:** Raise `self.retry(countdown=300)` immediately on -55. Do NOT write a position record. Do NOT decrement retry count — let Celery handle it.
**Warning signs:** Flood of `position=None` records for Yandex keywords shortly after first deploy.

### Pitfall 2: SOCKS5 ImportError at Health Check Time

**What goes wrong:** `httpx.Client(proxy="socks5://...")` raises `ImportError: install httpx[socks]` the first time a SOCKS5 proxy is health-checked — not at app startup.
**Why it happens:** socksio is lazy-imported by httpx only when a SOCKS5 URL is actually used.
**How to avoid:** Change `requirements.txt` to `httpx[socks]>=0.27,<0.28`. Also add a startup check in `proxy_health_service.py`: attempt `import socksio` and log a warning if missing.
**Warning signs:** All SOCKS5 proxies immediately show status "dead" after adding.

### Pitfall 3: XMLProxy Tasks on the Wrong Queue

**What goes wrong:** XMLProxy tasks run on `crawl` queue (Playwright worker), competing with browser crawl jobs and wasting Playwright processes.
**Why it happens:** `celery_app.py` routes `app.tasks.position_tasks.*` to `crawl`. XMLProxy logic added to `position_tasks.py` inherits that routing.
**How to avoid:** The XMLProxy helper `_check_via_xmlproxy()` is called from within `check_positions` which already runs on `crawl`. This is acceptable since it's a helper function not a separate task. If a separate XMLProxy task is created, explicitly route it to `default` queue.
**Warning signs:** Crawl jobs timing out during position check cycles.

### Pitfall 4: yandex_region NULL on Existing Sites

**What goes wrong:** Sites created before migration have `yandex_region=NULL`; `str(None)` passed as `lr` parameter yields no results or an error.
**Why it happens:** New column without server_default applies only to new rows; existing rows get NULL.
**How to avoid:** Migration must include `server_default="213"` on the `ADD COLUMN` statement. The model `default=213` only applies to in-Python object creation. Handle gracefully in service: `lr = site.yandex_region or 213`.
**Warning signs:** All existing sites return 0 positions after migration.

### Pitfall 5: Balance Fetch Slows Down Position Checks

**What goes wrong:** Balance check adds 1-3s per position check cycle when called inside the per-keyword loop.
**Why it happens:** Balance endpoint called once per keyword instead of once per cycle.
**How to avoid:** Fetch balance once at the start of `_check_via_xmlproxy()`, before the keyword loop. If balance is 0, exit immediately without processing any keyword.

### Pitfall 6: native_enum=False Omitted on Proxy Enum Columns

**What goes wrong:** Migration creates PostgreSQL TYPE `proxytype` and `proxystatus`. Downgrade fails because the types are referenced by the table.
**Why it happens:** SQLAlchemy's default `SAEnum` creates native PostgreSQL ENUM types.
**How to avoid:** Use `SAEnum(ProxyType, native_enum=False)` in the model. Use `sa.String(20)` in the migration column definition. See migration 0019 for the established pattern.
**Warning signs:** `alembic downgrade` fails with "type proxytype still referenced."

## Open Questions

1. **XMLProxy `groupby` parameter format**
   - What we know: CONTEXT.md says it mirrors Yandex XML API. The `groupby` value uses URL-encoded format in practice.
   - What's unclear: Whether XMLProxy uses `groupby=attr=d.mode=deep.groups-on-page=10` as a literal string or requires URL encoding.
   - Recommendation: Test with user's actual XMLProxy credentials in Wave 1; the value shown in Pattern 1 is the standard Yandex XML format. Add a smoke test that logs the raw XML response.

2. **rucaptcha.com `getBalance` response format**
   - What we know: v2 API uses `{"clientKey": "..."}`, returns `{"errorId": 0, "balance": float}` per fetched docs.
   - What's unclear: Whether `getBalance` is the exact endpoint name or `getTaskBalance`.
   - Recommendation: The existing anti-captcha.com code in `proxy_serp_service.py` already polls for task results — balance check is a new endpoint. Test against live account in integration test.

3. **XMLProxy task queue decision**
   - What we know: `position_tasks.*` routes to `crawl` queue. XMLProxy doesn't need Playwright.
   - What's unclear: Whether it's worth splitting `_check_via_xmlproxy` into a separate task on `default` queue, or acceptable to keep it in the `crawl` queue given the existing task structure.
   - Recommendation (Claude's discretion): Keep XMLProxy logic inside `check_positions` as a helper function (not a separate task) — avoids queue routing complexity. The `crawl` queue workers can handle pure HTTP calls; they just also happen to have Playwright available.
