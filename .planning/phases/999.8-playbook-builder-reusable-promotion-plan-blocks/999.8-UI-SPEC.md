## Spacing Scale

Declared values (multiples of 4 only). Matches existing project usage in base.html, kanban.html, crm/index.html:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (p-1 / gap-1) | Icon-to-label gaps, badge inner padding |
| sm | 8px (p-2 / gap-2) | Compact element spacing, button inner gap |
| md | 16px (p-4 / gap-4) | Default card padding, form row gap |
| lg | 24px (p-6 / gap-6) | Card header padding, section top margin |
| xl | 32px (mb-8) | Between major content blocks |
| 2xl | 48px | Page-level section breaks (not used in dense admin UI) |
| 3xl | 64px | Reserved — not used in this phase |

Exceptions:
- Tab bar buttons: `padding: 0.75rem 1.25rem` (12px / 20px) — matches crm/detail.html tab pattern exactly
- Sidebar child items: `padding: 0.4rem 0.75rem 0.4rem 2.5rem` — matches existing sidebar.html
- Playbook step row: `padding: 0.75rem 1rem` (12px / 16px) — tight checklist spacing
- Banner: `padding: 0.75rem 1.5rem` — full-width sticky, needs visual weight
- Touch target minimum: 44px height for primary action buttons (banner CTA buttons)

## Typography

Source: base.html (`font-family: system-ui, sans-serif`) + existing patterns from kanban.html (text-sm = 14px, text-lg = 18px, font-semibold = 600).

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px (text-sm) | 400 (normal) | 1.5 | Step descriptions, card body text, form labels |
| Label | 12px (text-xs) | 600 (font-semibold) | 1.4 | Category badges, step status labels, table headers |
| Heading | 18px (text-lg) | 600 (font-semibold) | 1.2 | Page h2 headings, card titles (matches existing .card h2 pattern) |
| Display | 20px (text-xl) | 600 (font-semibold) | 1.2 | Banner step title, playbook builder header only |

Rules:
- Maximum 4 sizes in use at once (declared above — enforced).
- Maximum 2 weights: 400 (normal) and 600 (semibold) — no 500, no 700 except existing `.btn-primary`.
- Markdown-rendered content (`summary_md`, `checklist_md`) inherits body 14px; `<h2>` inside markdown renders at heading size (18px/600); `<li>` at body size.
- Banner text: Display (20px/600) for step title; Body (14px/400) for "Шаг N/M · Playbook: {name}".

## Interaction Contracts

### Tab Switching (Project Playbook tab)

Follows existing CRM `switchTab()` JS pattern. No page reload. HTMX lazy-loads tab content on first click, caches thereafter. Active tab: `border-bottom: 2px solid #4f46e5`. Inactive tabs: `color: #6b7280; border-bottom: 2px solid transparent`.

### Drag-and-Drop in Builder

1. User grabs drag handle (grip icon, `.drag-handle` class).
2. Sortable.js animates row repositioning (animation: 150ms).
3. On `onEnd`: client sends `POST /ui/playbooks/{id}/steps/reorder` with `{order: [uuid, ...]}`.
4. Server responds 204 No Content or JSON with new positions.
5. Client shows toast "Порядок сохранён" (success, 3s).
6. On network error: toast "Ошибка сохранения порядка" (error) + page reload to revert visual state.

Keyboard alternative: ↑ / ↓ buttons per row. Each triggers `PUT /ui/playbooks/{id}/steps/{step_id}/move?direction=up|down`. HTMX response replaces `#steps-list` with re-rendered list. Keyboard-only users must be able to reorder entirely without mouse.

### Banner State Machine (D-16/D-17)

States: `hidden` → `active` → `hidden`.

Transition to `active`:
1. User clicks "Перейти к шагу" on a non-manual_note step.
2. JS saves `{project_playbook_id, step_id, project_id, opened_at: Date.now()}` to `sessionStorage.active_playbook_step`.
3. Browser navigates to target route (full page load).
4. `playbook_banner.js` runs on DOMContentLoaded: reads sessionStorage → if present, fetches `/api/playbook-step-active?step_id={id}` → populates banner → removes `hidden`.
5. `.main-content` gets `padding-top` increased by banner height (add class `has-banner` which applies `padding-top: calc(1.5rem + 52px)` via inline style or CSS class).

Transition to `hidden`:
- "Отметить выполненным": `hx-post "/api/project-playbook-steps/{id}/complete"` → on success JS clears sessionStorage, removes banner, redirects to `/ui/projects/{project_id}#playbook`.
- "Вернуться к плейбуку": JS clears sessionStorage, navigates to `/ui/projects/{project_id}#playbook` (no server call).
- If fetch to `/api/playbook-step-active` returns 404 (step was completed externally): clear sessionStorage silently, no banner shown.

Banner must NOT appear on mobile `/m/` routes. Banner JS checks `window.location.pathname.startsWith('/m/')` — if true, skips banner entirely.

### Step Status Toggle

Clicking the circle status icon cycles: `open` → `in_progress` → `done` → `open`.
HTMX POST to `/api/project-playbook-steps/{id}/status` with `{status: "next_status"}`.
Server returns a rendered partial of the step row (`hx-swap="outerHTML"`).
On blocked step status change (skipping prereq): server returns 200 with updated row that includes warning toast trigger header `HX-Trigger: {"showToast": {"msg": "...", "type": "warning"}}`.

### HTMX Filter (Block Library)

Both dropdowns on `/ui/playbooks/blocks` use:
```
hx-get="/ui/playbooks/blocks"
hx-trigger="change"
hx-target="#blocks-grid"
hx-push-url="true"
hx-include="[name='category_id'],[name='expert_source_id']"
hx-vals='{"partial": "true"}'
```
Loading indicator: `htmx-indicator` spinner inside `#blocks-grid`.

## Accessibility Contracts

### Keyboard Navigation

- Drag-and-drop builder: keyboard fallback via ↑/↓ buttons per row (mandatory, see Component Inventory §4).
- All icon-only buttons must have `aria-label` in Russian: e.g. `aria-label="Переместить блок выше"`, `aria-label="Удалить шаг"`.
- Tab bar buttons: `role="tab"`, `aria-selected="true/false"`, `aria-controls="tab-playbook"`.
- Banner: `role="alert"` (because it appears without user action on page load). `aria-live="polite"` on the banner container.
- Modal: `role="dialog"`, `aria-modal="true"`, `aria-labelledby="modal-title"`. Focus trapped inside while open. Close on Escape key.
- Status toggle circle buttons: `aria-label="Статус шага: {current_status}. Нажмите для изменения."`.

### Focus States

All interactive elements use existing Tailwind `focus:ring-indigo-500 focus:border-indigo-500` pattern (from kanban.html / crm/index.html). No custom focus overrides.

### ARIA Live Regions

- Toast container (`#toast-container`): existing `aria-live` not set — executor should add `aria-live="polite" aria-atomic="true"` to the toast container div in base.html.
- Hint badge for completed steps: `aria-label="Шаг, вероятно, выполнен. Нажмите для подтверждения."`.
