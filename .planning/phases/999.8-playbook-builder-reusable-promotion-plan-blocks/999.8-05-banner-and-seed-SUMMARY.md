# Phase 999.8 Plan 05: Banner and Seed Summary

**One-liner:** Global sticky indigo banner overlay (D-16/D-17) with sessionStorage-driven state, 2 supporting API endpoints, and an idempotent demo seed script — closing the walking-skeleton end-to-end loop for Phase 999.8.

---

## What Was Built

### 1. Banner API endpoints (`app/routers/playbooks.py`)

Appended under a new `# --- Banner Endpoints (Plan 05) ---` section at the bottom of the router to stay out of Plan 04's apply/hints territory. Both endpoints are authenticated but open to any team member with project access per D-20.

```
GET  /api/playbook-step-active?step_id={uuid}
POST /api/project-playbook-steps/{step_id}/complete
```

- `api_playbook_step_active` runs a single `ProjectPlaybookStep` query with `selectinload(block)` and `selectinload(project_playbook).selectinload(steps)` so the response can compute `position/total` without a second round trip. Returns 404 on missing step so the banner JS can silently clear stale `sessionStorage` state.
- `api_project_step_complete` sets `status = done` and stamps `completed_at` (only when previously NULL — idempotent re-clicks keep the original timestamp). Returns `{status, project_id}` so the banner JS can redirect back to `/ui/projects/{id}/kanban#playbook` without a second fetch.

Both use inline SQLAlchemy rather than `playbook_service` because this plan runs in parallel with Plan 04, and Plan 04 is the one adding the `get_project_step / set_step_status / get_project_playbook` service helpers. Keeping the queries inline guarantees the endpoints are verifiable in isolation.

### 2. Banner component (`app/templates/components/playbook_banner.html`)

Full-width sticky alert, hidden by default:

- `position:sticky; top:0; z-index:30` — stays glued to the viewport top above the main content
- `background:#4f46e5` (indigo-600) with white text — matches UI-SPEC D-16/D-17
- `role="alert"` + `aria-live="polite"` — announced by screen readers without yanking focus
- Two CTA buttons, both `min-height:44px` (touch-target minimum from UI-SPEC):
  - "Отметить выполненным" — white bg, indigo-700 text, primary weight
  - "Вернуться к плейбуку" — indigo-500 bg, indigo-400 border, secondary
- `<script src="/static/js/playbook_banner.js" defer>` wired at the bottom of the partial

### 3. Banner JS (`app/static/js/playbook_banner.js`)

IIFE with `'use strict'`. Single `init()` fires on `DOMContentLoaded`:

1. If `pathname` starts with `/m/` → `hide()` and return (mobile has its own base template, so skipping is cosmetic).
2. Read `sessionStorage.active_playbook_step`. If absent / malformed → `hide()` and silently clear.
3. `fetch('/api/playbook-step-active?step_id=…')`:
   - 404 → silently clear sessionStorage, hide banner
   - non-ok → hide banner
   - `data.status === 'done'` → clear and hide
   - else render `Шаг N/M · {title}` + `Playbook: {name}` and wire buttons
4. "Отметить выполненным" → `POST /api/project-playbook-steps/{id}/complete`:
   - ok → clear sessionStorage → `window.location.href = /ui/projects/{id}/kanban#playbook`
   - non-ok → russian alert "Не удалось отметить выполненным…"
5. "Вернуться к плейбуку" → clear sessionStorage → same redirect (no POST).

Uses `var`/function expressions throughout so it runs without transpilation on older browsers (consistent with the project's no-build CDN posture).

### 4. `base.html` include

Inserted `{% include "components/playbook_banner.html" %}` immediately before the breadcrumb include inside `.main-content`. Order verified by Python check:

- **6 PlaybookBlocks** — one per action_kind:

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Seed script crashed with NoReferencedTableError at first flush**

- **Found during:** Task 3, first run inside the api container.
- **Issue:** The script imported only `app.models.playbook`. SQLAlchemy's metadata therefore saw `playbook_blocks.created_by → users.id` and `project_playbooks.project_id → projects.id` FKs without the target `User` / `Project` mappers being registered, which raised `NoReferencedTableError: Foreign key associated with column 'playbook_blocks.created_by' could not find table 'users'`.
- **Fix:** Added `import app.models` (the package auto-imports most models), plus explicit `import app.models.user` and `import app.models.project` (those two are not re-exported from `app.models.__init__`). Seed now flushes cleanly.
- **Files modified:** `scripts/seed_demo_playbook.py`
- **Commit:** `e69ce44`

**2. [Rule 3 - Blocking] Plan-supplied session import was wrong**

- **Found during:** Task 3 drafting.
- **Issue:** The plan used `from app.db.session import async_session_factory` as a placeholder, but this project uses `app.database.AsyncSessionLocal` (confirmed by reading `scripts/seed_admin.py`, the existing seed-script reference).
- **Fix:** Used `from app.database import AsyncSessionLocal` and the `async with AsyncSessionLocal() as session` context-manager pattern established by `seed_admin.py`.
- **Files modified:** `scripts/seed_demo_playbook.py`
- **Commit:** `e69ce44`

**3. [Rule 2 - Missing] Plan 04 service helpers unavailable at parallel execution time**

- **Found during:** Task 1 drafting.
- **Issue:** The plan's suggested endpoint code called `svc.get_project_step`, `svc.set_step_status`, and `svc.get_project_playbook` — all helpers that Plan 04 was to add in the same wave. Because Plan 05 and Plan 04 run in parallel, there was no guarantee those helpers would exist when Task 1 committed.
- **Fix:** Rewrote both banner endpoints to use inline SQLAlchemy (`select(ProjectPlaybookStep).options(selectinload(...))`) — no dependency on Plan 04. The endpoints now pass their isolated acceptance test regardless of Plan 04's progress. Plan 04 did land its changes shortly after Task 1 (`ProjectPlaybookStepStatus` import + `compute_hints_for_playbook` import), and the combined module still imports cleanly (`app.main` = 459 routes).
- **Files modified:** `app/routers/playbooks.py`
- **Commit:** `9cf442c`

### Parallel Execution Coordination

- The `<parallel_execution>` block in the executor prompt explicitly required `--no-verify` on all commits and a `# --- Banner Endpoints (Plan 05) ---` header under the new section. Both honoured.
- Plan 04 added its own imports (`ProjectPlaybookStepStatus`, `compute_hints_for_playbook`) at the top of `app/routers/playbooks.py` concurrently; those appeared mid-task and were left untouched (they are additive and compatible with the banner section).

## End-to-End Walkthrough (post-Plan-05 walking skeleton)

Once both Plan 04 and Plan 05 are merged, the full loop works as follows (requires a running stack and `alembic upgrade head` + the demo seed):

1. `docker compose exec api python scripts/seed_demo_playbook.py` → 6 blocks + 1 playbook appear.
2. Admin logs in → sidebar "Плейбуки" → Library shows 6 Russian demo blocks.
3. Template list shows "SEO для коммерческих страниц (демо)" with `Опубликован` badge and "6 шагов" stat.
4. Admin applies the playbook to any existing Project from the new Project Playbook tab (Plan 04).
5. Clicking "Перейти к шагу" on step 1 writes `sessionStorage.active_playbook_step` and navigates to the target URL (resolved by Plan 04's `/api/playbook-step-route`).
6. Target page loads → `playbook_banner.js` reads sessionStorage → fetches `/api/playbook-step-active` → renders the indigo banner.
7. Clicking "Отметить выполненным" → `POST /api/project-playbook-steps/{id}/complete` → redirect back to `/ui/projects/{id}/kanban#playbook` with the step now marked done.
8. Visiting any other page (or `/m/*` mobile routes) shows no banner — sessionStorage was cleared on completion.
