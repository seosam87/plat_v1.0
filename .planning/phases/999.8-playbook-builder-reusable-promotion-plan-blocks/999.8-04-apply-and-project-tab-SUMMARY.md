# Phase 999.8 Plan 04: Apply and Project Tab Summary

**One-liner:** Wires playbook templates into the project experience with copy-on-apply + tabbed Kanban page + HTMX step checklist + completion-hint service, closing the walking skeleton minus the banner (which Plan 05 delivers in parallel).

---

## What Was Built

### 1. Service layer extensions (`app/services/playbook_service.py`)

Appended a new block of helpers for the applied-copy lifecycle without touching the existing block/template CRUD that Plans 02 and 03 own:

- `ACTION_KIND_TO_ROUTE: dict[str, str]` — the D-02 mapping from ActionKind enum values to UI route templates. All 7 action kinds covered (manual_note → empty string).
- `resolve_action_route(action_kind, project_id, site_id)` — templated `.format(site_id=..., project_id=...)` lookup returning `str | None`.
- `apply_playbook(db, project_id, playbook_id, applied_by, custom_name=None)` — copy-on-apply: creates one `ProjectPlaybook` and one `ProjectPlaybookStep` per source `PlaybookStep`, cloning `block.prerequisites` and `note_md`. Commits and returns the new `ProjectPlaybook`, or `None` if the source template is missing.
- `list_project_playbooks(db, project_id)` — eager-loads `steps → block → (category, expert_source, media)` via `selectinload` so the tab template can render without lazy-load warnings; filters out archived playbooks; ordered by `applied_at DESC`.
- `get_project_step(db, step_id)` — same eager-load chain + `project_playbook` relationship, used by status toggle + route resolution.
- `set_step_status(db, step_id, new_status)` / `cycle_step_status(db, step_id)` — 3-state cycle (open → in_progress → done → open). `completed_at` stamped on transition to `done`, cleared on transition back to `open`.
- `open_step_action(db, step_id)` — stamps `opened_at = now()` the first time it's called, and transitions `open → in_progress` so the UI immediately shows work-in-progress. Subsequent calls are no-ops.
- `update_project_step_note(db, step_id, note_md)` — blur auto-save for manual_note textareas.
- `get_project_playbook(db, pp_id)` + `archive_project_playbook(db, pp_id)` — archive admin action.
- `get_project_site_id(db, project_id)` — one-column lookup used by hints + route resolution.

### 2. Hint service (`app/services/playbook_hints.py`)

New file. Implements D-18 "Похоже, выполнено" heuristics for the Playbook tab:

- `check_step_hint(db, step, *, site_id, project_id) -> bool` dispatches on `step.block.action_kind.value`:
  - `run_crawl` → `exists(CrawlJob.site_id == site_id, status == done, started_at > step.opened_at)`
  - `open_keywords` → `exists(Keyword.site_id == site_id, created_at > step.opened_at)`
  - `open_competitors` → `exists(Competitor.site_id == site_id, created_at > step.opened_at)`
  - `open_content_plan` → `exists(ContentPlanItem.project_id == project_id, created_at > step.opened_at)`
  - `open_commercial_check` / `open_brief` → `return False` with `TODO(999.8.1)` comment (see Decisions)
  - `manual_note` / unknown → `return False`
- `compute_hints_for_playbook(db, steps, *, site_id, project_id) -> dict[step.id, bool]` — batches hint checks for one ProjectPlaybook's step list; skips in_progress/done steps (they already have explicit state).

All hint queries are cheap `SELECT EXISTS(...)` scoped by FK + `> step.opened_at`, so a playbook tab render adds at most `N_open_steps` tiny queries.

### 3. Router endpoints (`app/routers/playbooks.py`)

Appended under a new `# --- Apply + Hints + Step Route (Plan 04) ---` section, leaving Plan 02 (block CRUD), Plan 03 (builder), and Plan 05 (banner) territories untouched. 9 endpoints total:

| Method | Path | Auth | Purpose |
|---|---|---|---|
| GET | `/ui/projects/{project_id}/playbooks/apply-modal` | admin | Fragment for HTMX-loaded apply modal |
| GET | `/ui/playbooks/{playbook_id}/preview` | any auth | Small HTML fragment used by modal `<select>` onchange |
| POST | `/ui/projects/{project_id}/playbooks/apply` | admin | Copy-on-apply + HX-Redirect close |
| GET | `/ui/projects/{project_id}/playbook-tab` | any auth | Tab panel partial with `project_playbooks` + `hints` + `blocked` dicts |
| POST | `/api/project-playbook-steps/{step_id}/status` | any auth | Cycle status + return re-rendered row partial (outerHTML swap) |
| POST | `/api/project-playbook-steps/{step_id}/open-action` | any auth | Stamp `opened_at`, transition to `in_progress` |
| GET | `/api/playbook-step-route` | any auth | Walk step → project → site, call `resolve_action_route`, return `{url}` |
| POST | `/api/project-playbook-steps/{step_id}/note` | any auth | Blur-save `note_md` (204) |
| POST | `/ui/project-playbooks/{pp_id}/archive` | admin | Archive ProjectPlaybook, redirect to `#playbook` |

Blocked-prereq resolution happens server-side in `ui_project_playbook_tab`: a `block_id → status` map is built per ProjectPlaybook, then every step's `prerequisites` list (PlaybookBlock UUIDs) is resolved against that map. `blocked[step.id]` ends up as a dict the template reads without any Python in Jinja.

### 4. Kanban tabbed refactor (`app/templates/projects/kanban.html`)

In-place refactor adding a tab bar above the existing content:

- Tab bar markup from `crm/detail.html` (buttons with `role="tab"`, `aria-selected`, `aria-controls`, inline indigo-600 underline).
- Original kanban body (task columns, new-task-form, comments form, htmx-ext json-enc loader) wrapped verbatim in `<div id="tab-kanban" class="proj-tab-panel">` — no hx-patch or task-card markup touched.
- Second panel `<div id="tab-playbook">` with `hx-get="/ui/projects/{id}/playbook-tab" hx-trigger="intersect once" hx-target="this" hx-swap="innerHTML"` and a Russian loading placeholder.
- `switchProjectTab(name)` JS (separate class `.proj-tab` to avoid CSS bleed with CRM's `.crm-tab`) that toggles `aria-selected`, inline border-bottom color, and panel display.
- Auto-activation on page load if `location.hash` is `#playbook` or `#kanban` — enables the apply flow's HX-Redirect target.

### 5. Project playbook tab template (`app/templates/projects/_playbook_tab.html`)

- Empty state: centered heading "Плейбуки не применены" + admin-only "Применить плейбук" button that HTMX-loads the modal into `#apply-modal-container`.
- Populated state: admin-only action row with Apply button, then one card per `ProjectPlaybook`:
  - Header: name + status badge (Активен / На паузе / Завершён / Архив) with per-status color classes
  - Admin-only Архивировать form with `hx-confirm`
  - Progress bar: "Выполнено: N/M шагов" text + indigo-600 filled bar at `{pct}%`
  - Step list: `{% include "projects/_playbook_project_step.html" %}` inside a `{% with hint, is_blocked %}` scope

### 6. Step row partial (`app/templates/projects/_playbook_project_step.html`)

Renders one `ProjectPlaybookStep` with all states:

- **Row colors** (from UI-SPEC §5 color contracts):
  - open → bg-white, border-gray-100
  - in_progress → bg-indigo-50, border-indigo-200
  - done → bg-white, title text-gray-400 line-through
  - blocked → border-rose-200 + tooltip "Сначала завершите предыдущие шаги"
- **Status toggle button** — `hx-post` to `/api/project-playbook-steps/{id}/status` with `hx-swap="outerHTML"` and `hx-target="closest [data-step-id]"`. Circle icon changes by state: emerald-500 ✓ / indigo-500 ● / white outline ○.
- **Hint badge** — rendered only when `hint=True` and status ≠ done: amber-100/amber-700 pill "✓ Похоже, выполнено" + "Подтвердить" link that also hx-posts the status endpoint.
- **Blocked badge** — rose-100/rose-700 "Заблокирован" pill.
- **manual_note expansion** — `<details>` with `{{ step.block.summary_md | markdown }}` + optional checklist + media links + auto-save textarea (`hx-trigger="blur changed delay:300ms"`).
- **"Перейти к шагу" button** (non manual_note) — calls `openPlaybookStep(...)` JS helper.
- **openPlaybookStep JS** — defined once per page via `if (typeof window.openPlaybookStep === 'undefined')`, so outerHTML re-includes don't redefine it:
  1. Writes `{step_id, project_playbook_id, project_id, title, position, total, playbook_name, opened_at}` to `sessionStorage.active_playbook_step` (the Plan 05 banner reads this).
  2. Fires `POST /api/project-playbook-steps/{id}/open-action` fire-and-forget.
  3. Fetches `/api/playbook-step-route?step_id=...`, redirects to `data.url` if present, else alerts "Шаг завершается вручную" for manual_note.
  4. Russian error message on failure so the UI stays consistent.

### 7. Apply modal (`app/templates/playbooks/_apply_modal.html`)

`role="dialog"` / `aria-modal="true"` backdrop with click-outside + × close buttons. Body branches:

- No published playbooks → friendly empty state with "Сначала создайте и опубликуйте шаблон..." hint.
- Published playbooks → `<select>` + inline onchange handler that `fetch`es `/ui/playbooks/{id}/preview` and inlines the HTML into `#pb-preview` — gives the admin a description + step count + ETA before clicking Применить.
- Submit → `hx-post` to `/ui/projects/{id}/playbooks/apply` (HX-Redirect closes the modal and reloads to `#playbook`).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing] Status partial context was missing per-step hint/blocked values**

- **Found during:** Task 2 (router endpoint drafting, continued into Task 3 render verification)
- **Issue:** The plan's skeleton for `api_project_step_status` returned the row partial with `{"blocked": False, "hint": False}` scalars, but the template reads `hints.get(step.id)` and `blocked.get(step.id)` via the `{% with %}` block in `_playbook_tab.html`. For standalone swaps (status toggle), the include path receives `hint` and `is_blocked` directly from the outer `{% with %}`.
- **Fix:** Router endpoint now passes both the scalars (`hint=False`, `is_blocked` resolved lazily) AND dicts (`hints={step.id: False}`, `blocked_map={step.id: False}`) so both render paths work. The template uses `{% set status = step.status.value %}` + local `hint` / `is_blocked` variables which the caller supplies via `{% with %}` — standalone swaps land at the same namespace.
- **Files modified:** `app/routers/playbooks.py` (already committed in `7c4fad8` by previous executor)
- **Commit:** `7c4fad8`

**2. [Rule 3 - Blocking] _playbook_tab.html needed to dodge Jinja `{% with %}` with computed loop vars**

- **Found during:** Task 3 template drafting
- **Issue:** The plan suggested `{% with hint=hints.get(step.id, False), is_blocked=blocked.get(step.id, False) %}{% include ... %}{% endwith %}`. That's valid Jinja but puts `hint` and `is_blocked` in the inner include's namespace, which the partial must then reference directly (not via `hints[step.id]`). The partial already expected scalars, so the fix was to make sure both the partial and the tab template agree on the local names — which they now do.
- **Fix:** Final template reads `hint` and `is_blocked` as scalar locals inside the partial; no change needed to the inner loop beyond this documentation note.
- **Files modified:** none (caught during template drafting before commit)
- **Commit:** n/a

## End-to-End Walkthrough

Requires a running stack with `alembic upgrade head` and the Plan 05 demo seed (`scripts/seed_demo_playbook.py`):

1. Admin logs in → opens any project `/ui/projects/{id}/kanban`.
2. Sees tab bar: **Канбан** (active, indigo underline) | **Плейбуки** (inactive gray).
3. Clicks **Плейбуки** → HTMX `intersect once` fires → GET `/ui/projects/{id}/playbook-tab` → empty state rendered with "Плейбуки не применены" + admin-only Применить button.
4. Clicks **Применить плейбук** → HTMX GET `/ui/projects/{id}/playbooks/apply-modal` → modal appears with "SEO для коммерческих страниц (демо)" in the dropdown.
5. Selects the demo playbook → inline onchange fetch → `#pb-preview` fills with description + "6 шагов · ~12 дней".
6. Clicks **Применить** → hx-post `/ui/projects/{id}/playbooks/apply` → copy-on-apply runs → HX-Redirect response → browser reloads to `/ui/projects/{id}/kanban#playbook`.
7. Page loads with Playbook tab pre-activated (`#playbook` hash) → tab lazy-loads and renders one card: "SEO для коммерческих страниц (демо)" with 0/6 progress bar + 6 step rows.
8. Admin clicks the status circle on step 1 ("Первичный краулинг сайта") → HTMX POST cycles to `in_progress` → row re-renders with indigo-50 background + indigo border.
9. Admin clicks **Перейти к шагу** on step 1 → sessionStorage written → opened_at stamped → fetch to `/api/playbook-step-route` returns `/ui/sites/{site_id}/crawls` → browser navigates → Plan 05 banner overlay appears on the target page.
10. Admin runs a crawl → returns to Playbook tab → HTMX re-fetches the tab (via refresh or banner "Вернуться к плейбуку") → the run_crawl step shows amber "✓ Похоже, выполнено" hint badge because `CrawlJob.started_at > step.opened_at`.
11. Admin clicks **Подтвердить** on the hint badge → status cycles to done → emerald checkmark + strikethrough title.
12. Step 6 is a manual_note ("Обсудить стратегию с клиентом") → admin clicks **Развернуть** → details panel opens with rendered Markdown + textarea → types a note → blur auto-saves via hx-post.
