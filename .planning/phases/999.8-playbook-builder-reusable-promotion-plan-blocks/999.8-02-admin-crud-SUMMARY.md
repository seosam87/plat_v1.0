## Deviations from Plan

### Rule 3 ŌĆö Blocking issue auto-fixed

**1. [Rule 3 - Convention] Service uses `db.flush()` not `db.commit()`**
- **Found during:** Task 1
- **Issue:** The plan snippet called `await db.commit()` + `await db.refresh()` inside every service function. The existing `get_db` dependency in `app/dependencies.py` already commits the session after the handler returns (and rolls back on exception), so calling `commit()` inside the service would double-commit and break the transaction boundary relied on by other routers.
- **Fix:** Replaced every `await db.commit()` with `await db.flush()` in `playbook_service.py`. Matches the convention in `template_service.py` and `user_service.py`.
- **Files modified:** `app/services/playbook_service.py`
- **Commit:** `5b33916`

### Rule 2 ŌĆö Auto-added missing critical functionality

**2. [Rule 2 - Error handling] Catch Pydantic ValidationError from `_parse_media_rows`**
- **Found during:** Task 3 smoke test
- **Issue:** The plan specified that `BlockMedia.url` should be validated via Pydantic `AnyHttpUrl` (CD-09) and that "Invalid URL in BlockMedia form ŌåÆ Pydantic AnyHttpUrl rejects ŌåÆ 422 or form re-render with error". But the router code constructed `BlockMediaCreate(...)` directly inside `_parse_media_rows`, outside FastAPI's request-validation layer. A bad URL raised `ValidationError` which escaped as a 500 through the middleware stack (confirmed in smoke test).
- **Fix:** Wrapped the `_parse_media_rows(form)` call in both POST handlers (`ui_playbook_block_create`, `ui_playbook_block_update`) with `try / except ValidationError` and re-raise as `HTTPException(status_code=422, detail="ąØąĄą║ąŠčĆčĆąĄą║čéąĮčŗą╣ URL ą▓ čüčüčŗą╗ą║ą░čģ: ŌĆ”")`. Added `from pydantic import ValidationError` and `status` import.
- **Files modified:** `app/routers/playbooks.py`
- **Commit:** `b2a1f68` (bundled with template commit)
- **Re-verified:** Re-ran smoke test ŌåÆ `invalid url status: 422` with Russian message.
