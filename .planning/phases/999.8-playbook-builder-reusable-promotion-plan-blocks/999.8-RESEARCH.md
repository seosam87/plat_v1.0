### Markdown dependency (CD-01 resolution)

`python-markdown` is NOT in `requirements.txt`. It is NOT installed in the current environment (verified: `import markdown` → ImportError). However, `app/routers/channel.py` line 340 does a guarded `import markdown` with a fallback, meaning the intent was to use `python-markdown`.

**Resolution CD-01:** Install and use **`python-markdown`** (`Markdown>=3.6,<4.0`). Rationale:
1. `channel.py` already has `import markdown` — same library, zero new ecosystem risk.
2. `app/main.py` line 2116 has a custom `_render_markdown()` function for the help system — this exists precisely BECAUSE `python-markdown` wasn't in requirements yet. Once installed, the help renderer could be simplified (but that's out of scope for 999.8).
3. `bleach` for XSS sanitization: `python-markdown` does not sanitize HTML by default. Given this is an admin-only UI (D-20: only admins create/edit block content), XSS risk is low. Use `extensions=["nl2br", "tables"]` without `bleach` — consistent with the existing `channel.py` usage.

Add to `requirements.txt`: `Markdown>=3.6,<4.0`

### ORM query skeleton for `playbook_hints.py`

```python
from sqlalchemy import select, exists
from sqlalchemy.ext.asyncio import AsyncSession
from app.models.crawl import CrawlJob, CrawlJobStatus
from app.models.keyword import Keyword
from app.models.competitor import Competitor
from app.models.content_plan import ContentPlanItem
from app.models.commerce_check_job import CommerceCheckJob
from app.models.brief_job import BriefJob

async def check_step_hint(
    db: AsyncSession,
    step: "ProjectPlaybookStep",
    project_site_id: uuid.UUID,
    project_id: uuid.UUID,
) -> bool:
    if step.opened_at is None:
        return False
    kind = step.block.action_kind.value   # assumes block is loaded

    if kind == "run_crawl":
        q = select(exists().where(
            CrawlJob.site_id == project_site_id,
            CrawlJob.status == CrawlJobStatus.done,
            CrawlJob.started_at > step.opened_at,
        ))
    elif kind == "open_keywords":
        q = select(exists().where(
            Keyword.site_id == project_site_id,
            Keyword.created_at > step.opened_at,
        ))
    elif kind == "open_competitors":
        q = select(exists().where(
            Competitor.site_id == project_site_id,
            Competitor.created_at > step.opened_at,
        ))
    elif kind == "open_content_plan":
        q = select(exists().where(
            ContentPlanItem.project_id == project_id,
            ContentPlanItem.created_at > step.opened_at,
        ))
    elif kind == "open_commercial_check":
        # No project/site scope — time-based global check
        q = select(exists().where(
            CommerceCheckJob.status == "complete",
            CommerceCheckJob.completed_at > step.opened_at,
        ))
    elif kind == "open_brief":
        q = select(exists().where(
            BriefJob.status == "complete",
            BriefJob.completed_at > step.opened_at,
        ))
    else:
        return False

    result = await db.execute(q)
    return result.scalar()
```

**Scope estimate:** 7 action_kinds, 6 branches (~80–100 lines including imports, docstring, error handling). Fits in one plan task. Scope watch from D-18 says "if > 1 plan, defer half" — at ~80 lines it is one plan task, not multiple.

## Open Questions for Planner

1. **`open_commercial_check` and `open_brief` hint scoping**: Both `CommerceCheckJob` and `BriefJob` have only `user_id` FK — no `site_id` or `project_id`. The hint query currently cannot be scoped to a specific project or site. Options:
   - Use time-only filter (any completed job after `opened_at`) — may produce false positives if multiple projects are active
   - Skip these two hint types (always return `False`) and add note to D-18 scope watch
   - **Planner recommendation needed:** Accept imprecise hint for these two, or omit. The hint is non-blocking (D-18), so false positives have low cost.

2. **Project detail page structure**: The current project "detail page" is `/ui/projects/{id}/kanban` — a standalone page, not a tabbed page. Adding the Playbook tab (D-14) requires either:
   - Refactoring `kanban.html` to a tabbed layout with Content Plan as another tab (3 tabs: Kanban, Content Plan, Playbook)
   - Creating a new `/ui/projects/{id}` route that renders a tabbed page, with `/kanban` and `/plan` as HTMX-loaded tab content
   The second option is cleaner but means more routes. The first option reuses existing routes but changes the Kanban page structure. The existing `plan.html` template would need to become a partial. Planner should decide.

3. **`ProjectPlaybook.project_id` → `Project.site_id` join for hints**: The hint service needs `project.site_id` given `project_playbook.project_id`. This requires an additional DB fetch. Planner should ensure the `check_step_hints` call loads the `Project` record (or passes `site_id` directly).

4. **`app/navigation.py` and URL prefix for `_HELP_MODULE_MAP`**: The new `/ui/playbooks/` prefix should be added to `_HELP_MODULE_MAP` in `app/template_engine.py` for context-sensitive help. Add: `"/ui/playbooks": "playbooks"` and create `app/templates/help/playbooks.md` (or skip help for MVP).

5. **`ProjectPlaybookStep.prerequisites` after reorder (D-12)**: The prerequisites field contains UUIDs of `PlaybookBlock` IDs (from D-03) — NOT `ProjectPlaybookStep` IDs. When steps are reordered in `ProjectPlaybook`, prerequisites still reference the original block UUIDs, which remain stable. This means prerequisite validation at display time must resolve block UUIDs to the corresponding `ProjectPlaybookStep` in this specific `ProjectPlaybook`. The service layer must implement this lookup.
