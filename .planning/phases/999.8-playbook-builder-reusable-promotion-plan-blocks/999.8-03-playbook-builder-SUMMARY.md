# Phase 999.8 Plan 03: Playbook Builder Summary

**One-liner:** Drag-and-drop playbook template builder with Sortable.js@1.15.0 pinned inline, keyboard-accessible up/down fallback, JSON-body reorder endpoint, deep clone-with-(копия) suffix, and a two-panel library+steps layout — extending playbook_service with 11 Playbook/PlaybookStep CRUD functions behind the `# --- Playbook Template CRUD ---` banner so Plan 02 can safely append its own block CRUD in parallel without merge conflicts.

## What Was Built

### 1. Service layer — `app/services/playbook_service.py` (append section)

Appended after Plan 02's `# --- Block / Expert / Category CRUD ---` banner, under a new `# --- Playbook Template CRUD ---` banner. Imports extended to include `Playbook`, `PlaybookStep`, `PlaybookCategory` from `app.models.playbook`.

**New Pydantic schemas:**
- `PlaybookCreate` — name, description_md, category, is_published
- `PlaybookUpdate` — identical shape (slug never changes)

**New async functions (all take `db: AsyncSession` first):**

| Function | Purpose |
|---|---|
| `list_playbooks(db, *, category, only_published)` | Grid query with selectinload(steps→block), filter by category |
| `get_playbook_with_steps(db, id)` | Fetch with nested selectinload for block→category + block→expert_source |
| `create_playbook(db, data, created_by)` | Slug-unique create (uses `_unique_slug` from Plan 02) |
| `update_playbook(db, id, data)` | Update name/category/is_published/description |
| `delete_playbook(db, id)` | CASCADE drops steps via FK |
| `add_step(db, playbook_id, block_id)` | Append at `max(position)+1` using SQL aggregate; returns eager-loaded step |
| `remove_step(db, step_id)` | Delete + two-phase position re-compact |
| `reorder_steps(db, playbook_id, step_ids_in_order)` | Full permutation via two-phase `+1000` → final update |
| `move_step(db, playbook_id, step_id, direction)` | Keyboard up/down; delegates to `reorder_steps` |
| `update_step_note(db, step_id, note_md)` | Blur auto-save of per-step markdown note |
| `clone_playbook(db, id, created_by)` | Deep copy: new Playbook with `{name} (копия)`, `is_published=False`, all PlaybookStep rows cloned |

**Critical correctness fix (Rule 1 during execution):**

`add_step` originally used `max((s.position for s in pb.steps), default=-1) + 1` where `pb.steps` came from `get_playbook_with_steps` eager load. SQLAlchemy's session identity map caches the `Playbook.steps` collection, so on 2nd+ consecutive calls it returned a stale snapshot (missing the newly added step), causing `uq_playbook_step_position` duplicate-key violations. Fixed by computing next position via a direct `SELECT func.max(PlaybookStep.position)` aggregate — always reflects current persisted state regardless of session cache.

Also replaced the exists-check with a cheap PK lookup (`SELECT Playbook.id WHERE id = :id`) so `add_step` no longer needs to eager-load the steps collection at all.

## Deviations from Plan

### Rule 1 — Bugs auto-fixed

**1. [Rule 1 — Bug] add_step position computation used stale session cache**
- **Found during:** Task 1 post-commit E2E smoke test (3 consecutive add_step calls)
- **Issue:** `max((s.position for s in pb.steps), default=-1) + 1` where `pb.steps` came from `get_playbook_with_steps()` — SQLAlchemy session identity map cached the `Playbook.steps` collection, so the 2nd `add_step` call saw the pre-1st-add snapshot and tried to insert at position 0 again, tripping `uq_playbook_step_position`.
- **Fix:** Compute `next_pos` via `SELECT func.max(PlaybookStep.position) WHERE playbook_id = :id` — SQL aggregates always reflect current persisted state regardless of ORM cache. Also replaced exists-check with cheap PK-only lookup to avoid eager loading at all.
- **Files modified:** `app/services/playbook_service.py`
- **Commit:** `ced8fce`
- **Verification:** Re-ran full smoke suite (create + 3 adds + reverse reorder + move_step up + remove_step + clone) → all positions correct, no duplicate-key errors.

**2. [Rule 1 — Bug] Template `|sum(attribute='block.estimated_days')` crashes on None**
- **Found during:** Task 3 post-commit template render smoke test with a fixture block having `estimated_days=None`
- **Issue:** `PlaybookBlock.estimated_days` is nullable; Jinja `|sum` raises `TypeError: unsupported operand type(s) for +: 'int' and 'NoneType'` when summing a mixed list.
- **Fix:** Replaced with a `{% set ns = namespace(total_days=0) %}` fold that coalesces `s.block.estimated_days or 0` inside the loop.
- **Files modified:** `app/templates/playbooks/_template_card.html`
- **Commit:** `e21ca17`
- **Verification:** Re-rendered `_template_card.html` with a fixture having `estimated_days=None` → 1321 chars, "Клонировать" present, no exception.
