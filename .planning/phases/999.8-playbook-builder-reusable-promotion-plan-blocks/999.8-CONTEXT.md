# Phase 999.8: Playbook Builder — Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Инструмент сборки плана SEO-работ для проекта из переиспользуемых «кубиков» (`PlaybookBlock`). Каждый кубик описывает тип работ, привязан к источнику компетенций (`ExpertSource`: Шестаков, Шакин, агентство) и к медиа-материалам (ссылки на видео/статьи — только URL, без хранения файлов и без транскриптов). `Playbook` — переиспользуемый шаблон из упорядоченных блоков. `ProjectPlaybook` — применённая копия шаблона с живым чеклистом прогресса. Шаги playbook ведут пользователя через существующие экраны системы (крауль, ключи, конкуренты, коммерческие страницы, бриф) через плашку сверху страницы.

**In scope (walking skeleton MVP):**
- Модели: `ExpertSource`, `BlockCategory`, `PlaybookBlock`, `BlockMedia`, `Playbook`, `PlaybookStep`, `ProjectPlaybook`, `ProjectPlaybookStep`
- Alembic-миграция для всех восьми таблиц
- Admin-CRUD для `PlaybookBlock` / `ExpertSource` / `BlockCategory`
- Drag-and-drop (или up/down) билдер шаблонов `Playbook` из блоков
- Применение `Playbook` к `Project` с созданием `ProjectPlaybook` и копий шагов
- Вкладка «Playbook» на странице проекта рядом с Kanban / Tasks / Content Plan (HTMX-swap)
- Живой чеклист шагов с ручным переключением статусов и текстовыми заметками
- Плашка-overlay поверх целевых страниц системы при «переходе к шагу»
- Новая sidebar-секция «Playbooks» с пунктами (Библиотека блоков, Шаблоны)

**Out of scope (deferred to phase 999.8.1 or v5.0):**
- Отдельные страницы экспертов `/ui/experts/{slug}` с агрегацией всех блоков эксперта
- Full-text search (PostgreSQL `tsvector`) по `summary_md` и `checklist_md`
- Хранение транскриптов видео (решено: только ссылки)
- Полноценный авто-детект завершения шага (остаётся только hint-badge, см. D-13)
- Версионирование `Playbook` и журнал изменений
- Аналитика по playbook'ам (% выполнения, среднее время на шаг)
- Клиент-видимый режим (все операции — admin/PM only в MVP)
- `PlaybookBlock.prompt_template_id` — добавляется в фазе 999.9 (Prompt Library)
- Статусы шага `blocked` / `skipped`

## Implementation Decisions

### Scope & Sizing
- **D-01:** Walking skeleton MVP — в фазе 999.8 полностью реализуются P1 (модели + админ-CRUD), P2 (билдер шаблонов), P3 (применение к проекту + чеклист) и упрощённая часть P4 (`ExpertSource` как справочник + `BlockMedia` как ссылки). Expert-страницы и FTS — следующая фаза (999.8.1 или начало v5.0).
- **Why:** Пользователь подтвердил walking skeleton — нужен end-to-end flow, который даёт немедленную ценность. Меньшие варианты (только модели, или модели + билдер без применения) не решают первоначальную задачу «построить план и провести через интерфейсы». Полный P1-P4 в одной фазе — 10+ планов, сложно верифицировать.

### PlaybookBlock Model

- **D-02:** `action_kind` — строгий enum без свободных URL. Значения MVP: `run_crawl`, `open_keywords`, `open_competitors`, `open_content_plan`, `open_commercial_check`, `open_brief`, `manual_note`. Расширяется добавлением значений в enum + случая в статическом dict-мапере `ACTION_KIND_TO_ROUTE` (в Python-коде, не в БД). Система сама подставляет `{project_id}` из контекста ProjectPlaybook — admin не вводит URL вручную.
- **Why:** Безопасность (никто не впишет внешний URL или XSS), простота валидации, автоматическое разрешение контекста (project_id), чёткая проверка прав доступа перед переходом. Расширение ценой PR с кодом — приемлемо, т.к. список работ конечен и медленно меняется.

- **D-03:** `prerequisites` — JSON-массив UUID блоков из того же `Playbook`. При применении playbook к проекту prerequisites копируются в `ProjectPlaybookStep.prerequisites`. Шаг со статусом `open` показывается как заблокированный, если хотя бы один prereq-шаг не `done`. Блокировка — только UI-hint, не hard-lock (PM может перевести шаг вручную, игнорируя prereq, — тогда показываем warning-тоast).
- **Why:** Пользователь выбрал вариант «Да, prerequisites — массив ID», несмотря на сложность. Массив проще отдельной join-таблицы при операциях copy-on-apply. Мягкая блокировка позволяет не ломать рабочий процесс, если методология позволяет параллельные шаги.

- **D-04:** `estimated_days` — целочисленное поле `Integer, nullable`. Сумма по шагам дает оценку playbook'а, показывается на странице применения к проекту как «ориентировочный срок: ~N дней». Не участвует в авто-расчёте дедлайнов — только справочно.
- **Why:** Пользователь подтвердил — нужно для оценок клиенту. Одно поле, минимальный impact на миграцию.

- **D-05:** `summary_md` (Text, required) — описание сути блока в Markdown. `checklist_md` (Text, nullable) — Markdown-список подзадач. Оба рендерятся через Jinja2 filter `| markdown` на базе `python-markdown` или `markdown-it-py` (выбор — Claude's Discretion). Чеклист отображается в плашке-overlay и в inline-раскрытии manual_note шага.
- **Why:** Два раздельных поля дают гибкое форматирование и семантически разделяют «что это за шаг» и «какие подзадачи выполнить». JSON-массив (3-й вариант) был отклонён — избыточно для MVP, Markdown достаточен.

### PlaybookBlock — подразумеваемые поля (не обсуждались явно, но следуют из решений)

Поля, которые планировщик должен включить в модель:

- `id: UUID PK`
- `title: str (required, max 500)`
- `slug: str (unique, generated from title)`
- `category_id: UUID FK → block_categories.id (required)`
- `expert_source_id: UUID FK → expert_sources.id (nullable — допускаются «общие» блоки без эксперта)`
- `summary_md: Text (required)`
- `checklist_md: Text (nullable)`
- `action_kind: Enum ActionKind (required)`
- `prerequisites: JSON array of UUIDs (default [])`
- `estimated_days: Integer (nullable)`
- `display_order: Integer (default 0, for library listing)`
- `created_at`, `updated_at`, `created_by` (FK users.id)

### ExpertSource, BlockCategory, BlockMedia

- **D-06:** `ExpertSource` — минимальная модель в MVP: `id`, `name`, `slug` (unique), `bio_md` (Text, nullable, краткое описание эксперта в Markdown), `external_url` (nullable, личный сайт/YouTube-канал), `created_at`. **Без** аватара, социальных ссылок, тегов — это добавляется в фазе 999.8.1 вместе с expert-страницами.
- **Why:** MVP цель — «к блоку приколот эксперт», а не «каталог экспертов как продукт». Минимум полей ускоряет миграцию и CRUD.

- **D-07:** `BlockCategory` — отдельная таблица (не enum): `id`, `name`, `slug`, `icon` (string, nullable — Tailwind heroicon name), `display_order`, `created_at`. Seed'ится начальным набором: «Анализ конкурентов», «Сбор ключей», «Анализ структуры сайта», «Коммерческие страницы», «Контент», «Техничка», «Ссылочная стратегия», «Регулярные работы».
- **Why:** Справочник категорий должен быть расширяемым через админку без миграций — поэтому таблица, не enum. Seed-миграция задаёт начальные восемь категорий из плана обсуждения.

- **D-08:** `BlockMedia` — `id`, `block_id FK`, `kind` enum (`video`, `article`), `url` (required, validated as URL), `title` (required), `description_md` (nullable, краткое описание под ссылкой), `display_order`, `created_at`. Полей `transcript_text` и `file_upload_id` **нет** — пользователь подтвердил вариант A (только ссылки). Список медиа на блок неограничен, отображается упорядоченным списком на карточке блока и в раскрывающейся панели шага.
- **Why:** Решение по транскриптам было зафиксировано в предыдущем обсуждении: «A — только ссылки». Нет S3, нет Whisper, нет парсера YouTube-субтитров. Минимум ценности при минимуме инфраструктуры.

### Playbook & PlaybookStep (шаблон)

- **D-09:** `Playbook` — шаблон плана: `id`, `name`, `slug`, `description_md` (nullable), `category` (enum коммерция/контент/технический/смешанный — опциональный тег для фильтра в списке), `is_published` (bool, default true — неопубликованные скрыты от PM при применении), `created_by`, `created_at`, `updated_at`. `PlaybookStep` — junction: `id`, `playbook_id FK`, `block_id FK`, `position` (Integer, unique per playbook), `note_md` (nullable — опциональная заметка «зачем именно в этом playbook»), `created_at`.
- **Why:** `Playbook` = переиспользуемый шаблон (подтверждено пользователем ранее: «playbook переиспользуется»). `PlaybookStep` — промежуточная таблица, позволяет одному блоку быть включённым в несколько playbook'ов с разными позициями и заметками.

### ProjectPlaybook & ProjectPlaybookStep (применение)

- **D-10:** `ProjectPlaybook` — применённая копия: `id`, `project_id FK`, `playbook_id FK` (исходный шаблон, для трассировки), `name` (копируется из шаблона, редактируемо), `status` enum (`active`, `paused`, `done`, `archived`), `applied_at`, `applied_by`, `updated_at`. **Один проект — много ProjectPlaybook** (1-to-many), несколько могут быть одновременно в статусе `active`.
- **Why:** Пользователь подтвердил «Несколько (Recommended)». Это даёт возможность вести на одном проекте «Коммерция от Шестакова» + «Регулярный контент от Шакина» параллельно.

- **D-11:** `ProjectPlaybookStep` — копия шага внутри применённого playbook: `id`, `project_playbook_id FK`, `block_id FK` (на исходный PlaybookBlock — для доступа к summary_md / checklist_md / action_kind), `position` (Integer, editable после apply), `status` enum (`open`, `in_progress`, `done`), `prerequisites` (JSON array, копия из PlaybookBlock.prerequisites, пересчитывается при reorder), `assignee_user_id` (nullable FK), `note_md` (nullable — свободная заметка PM о том, что сделано), `opened_at` (nullable — timestamp первого перехода к шагу, нужен для hint-триггера D-13), `completed_at` (nullable), `created_at`, `updated_at`.
- **Why:** Статусы 3-state (open/in_progress/done) достаточны для MVP. `blocked` выводится UI-логикой из prerequisites, `skipped` пока не нужен. `opened_at` критичен для гибридной completion-логики (D-13).

### ProjectPlaybook — связь и редактируемость

- **D-12:** `ProjectPlaybook` **полностью редактируется** после применения: PM может добавлять новые шаги (выбирая из библиотеки блоков), удалять, менять порядок (drag-and-drop), редактировать `note_md` и `assignee_user_id`. Исходный `Playbook`-шаблон **не затрагивается** — работает паттерн copy-on-apply. При изменении шаблона существующие `ProjectPlaybook` не синхронизируются.
- **Why:** Пользователь выбрал «Да, полностью (Recommended)». Без этой гибкости playbook становится жёстким и не соответствует реальному процессу SEO-работы, где методология адаптируется под сайт.

- **D-13:** `ProjectPlaybookStep` **не связан FK** с `SeoTask` / `ContentPlanItem`. Ручная связь через `note_md` (текстовая ссылка) или через будущие soft-ссылки — deferred. Выбор подтверждён пользователем: «Независимо (Recommended)».
- **Why:** Попытка жёсткой интеграции статусов двух систем создаёт дубликат источников истины и кошмарит верификацию. Playbook — уровень методологии, SeoTask — уровень атомарной задачи; разные гранулярности не должны цепляться FK.

### UI Placement

- **D-14:** Playbook в контексте проекта отображается как **новая вкладка** «Playbook» на странице `/ui/projects/{id}` рядом с существующими Kanban / Tasks / Content Plan. Переключение — HTMX-swap в контейнере табов. **Отдельного URL-а** `/ui/projects/{id}/playbook` в MVP нет — если появится необходимость делиться ссылкой, добавится позже.
- **Why:** Пользователь выбрал вкладку. Соответствует существующему паттерну страницы проекта (уже 3 вкладки). Не уводит из контекста проекта. HTMX-swap без отдельного URL минимизирует новые роуты.

- **D-15:** Новая sidebar-секция **«Playbooks»** (параллельно CRM) с подпунктами: «Библиотека блоков» (`/ui/playbooks/blocks`), «Шаблоны» (`/ui/playbooks`), «Эксперты» (admin-only, в MVP — упрощённый список без отдельных страниц). Точное расположение в sidebar — Claude's Discretion.
- **Why:** Phase 20 D-05 установил паттерн «новая секция для кластера фич». Playbook Builder — самостоятельный кластер, не подсекция CRM и не SEO Tools.

### Переход к шагу (Guide UX)

- **D-16:** При нажатии «Перейти к шагу» из чеклиста ProjectPlaybook открывается целевой URL (вычисленный из `action_kind`) + сверху страницы рендерится **full-width плашка-banner** в стиле Tailwind Alert/Notification: цветной фон (индиго), белый текст, структура «Шаг N/M · {block.title} · Playbook: {project_playbook.name}». Кнопки справа: `[Отметить выполненным]` (primary) и `[Вернуться к playbook]` (secondary). Состояние плашки — `{project_playbook_id, step_id, opened_at}` — хранится в `sessionStorage` под ключом `active_playbook_step`.
- **Why:** Пользователь выбрал «Плашка сверху (Recommended)». Shepherd.js отклонён — требует авторинга YAML для каждого action_kind (7 сценариев сразу), это раздувает MVP. sessionStorage переживает переход между страницами без query-параметров и не мусорит в URL.

- **D-17:** **Как плашка рендерится на всех страницах:** в `app/templates/base.html` добавляется один `{% include "components/playbook_banner.html" %}` сразу после breadcrumbs. Include содержит `<div id="playbook-banner" hidden>` + маленький `<script>` блок (inline в base.html или в `static/js/playbook_banner.js`), который:
  1. Читает `sessionStorage.getItem('active_playbook_step')`.
  2. Если есть — fetch к `/api/playbook-step-active` с `step_id` → получает title, position, total, playbook name.
  3. Рендерит плашку и показывает её.
  4. Кнопка «Отметить выполненным» → HTMX `POST /api/project-playbook-steps/{id}/complete` → при успехе очищает sessionStorage и переходит обратно на вкладку playbook проекта.
  5. Кнопка «Вернуться к playbook» → очищает sessionStorage и переходит на `/ui/projects/{project_id}` (с hash `#playbook` для авто-открытия вкладки).
- **Why:** Один include в base.html — zero per-template work (аналогично решению «Show Tour» из Phase 19.2 D-06). Клиентский JS минимален, никакой зависимости от Shepherd или Sortable на этой части.

### Completion Logic

- **D-18:** **Гибридная completion:** ручная кнопка «Отметить выполненным» всегда работает. Дополнительно — hint-badge: при открытии вкладки Playbook проекта backend проверяет для каждого `step.status == open` и его `action_kind`, произошло ли релевантное событие **после** `step.opened_at`. Если да — рисуем badge `✓ Похоже, выполнено` рядом с шагом и однокликовое «Подтвердить».
  - Mapping действий на триггеры (MVP, расширяемо):
    - `run_crawl` → последний `CrawlJob.status==done` для `project.site_id` с `started_at > step.opened_at`
    - `open_keywords` → новая запись в `keyword` таблице для проекта/сайта после `step.opened_at`
    - `open_competitors` → новая запись в `competitor` таблице
    - `open_content_plan` → новый `ContentPlanItem` в проекте
    - `open_commercial_check` → новый `commerce_check_job` с `status==done`
    - `open_brief` → новый `brief_job` с `status==done`
    - `manual_note` → нет триггера (всегда ручное)
  - Реализация — отдельный helper `services/playbook_hints.py` с функцией `check_step_hints(step) -> bool`. Вызывается только при загрузке страницы вкладки Playbook (не в фоне, не в Celery — синхронный SQL).
- **Why:** Пользователь выбрал «Гибрид», не рекомендованный «Вручную». Компромисс: ручная работает всегда (предсказуемо), hint не блокирует и не меняет статус сам — только подсказывает. Это не ломает D-13 (нет FK-интеграции), т.к. проверки — чистое чтение фактов существования в других таблицах, без двусторонней связи.
- **Scope watch:** 7 action_kind → 6 проверок → ~60 строк кода в `services/playbook_hints.py`. Если во время планирования оценка выйдет за 1 план — перенести half-проверок в 999.8.1 и оставить только `run_crawl` + `open_keywords` в MVP.

### manual_note блоки

- **D-19:** Для блоков с `action_kind == manual_note` кнопка «Перейти к шагу» не показывается. Вместо этого в чеклисте playbook'а шаг **раскрывается inline**: рендерится `summary_md`, `checklist_md`, ссылки из `BlockMedia`, и поле `note_md` становится редактируемой `<textarea>` с HTMX-save по blur. Статус меняется той же кнопкой «Отметить выполненным» рядом.
- **Why:** Пользователь выбрал «Остаёмся в чеклисте (Recommended)». manual_note — для теоретических/офлайновых задач («обсудить стратегию с клиентом»), где переход в систему бессмысленен.

### Access Control

- **D-20:** Create/Edit/Delete `PlaybookBlock`, `ExpertSource`, `BlockCategory`, `Playbook` — admin only (`require_admin`). View библиотеки блоков, шаблонов playbook'ов — любой аутентифицированный пользователь. Apply `Playbook` к `Project` — admin или PM (`require_admin` для MVP, `require_admin_or_manager` в будущем). Редактирование `ProjectPlaybookStep` (смена статуса, заметки) — любой пользователь с доступом к проекту.
- **Why:** Соответствует паттерну Phase 20 / 22. Библиотека знаний — редакторская работа, применение — менеджерская, исполнение — любой член команды.

### Claude's Discretion

- **CD-01:** Выбор Markdown-рендерера — `python-markdown` (проще, yes) vs `markdown-it-py` (быстрее, современнее). Планировщик решает по импорт-графу текущего проекта (если уже используется одна из библиотек — берём её).
- **CD-02:** Билдер playbook'а UI — Sortable.js через CDN (drag-and-drop) vs простая форма с кнопками `↑`/`↓`. Sortable.js — приятнее UX, но +1 CDN зависимость. Up/down кнопки — проще, работает без JS. **Предпочтение — Sortable.js**, если кол-во шагов в типичном playbook ≥ 5. Планировщик оценивает и выбирает.
- **CD-03:** Точное разбиение на файлы моделей — один `app/models/playbook.py` со всеми 8 классами vs отдельные файлы `expert_source.py`, `playbook_block.py` и т.д. Смотреть на существующие паттерны (Phase 22 использовал отдельный `proposal_template.py`, Phase 20 — отдельный `client.py`).
- **CD-04:** Структура Alembic-миграции — одна большая миграция со всеми таблицами или две (справочники + playbook-сущности). **Предпочтение — одна миграция**, все таблицы связаны и должны появляться атомарно.

<specifics>
## Specific Ideas

- **«Источник компетенций» — основной дифференциатор фичи.** Карточка `PlaybookBlock` должна показывать имя эксперта крупно (avatar-less pill/badge с slug), его отсутствие — редкий случай. Админский CRUD блока требует выбрать `ExpertSource` или явно отметить «Общий».
- **Плашка шага — magic-moment фичи.** Её визуальная заметность важнее «красоты»: цветной фон, sticky top, не перекрывается с breadcrumbs. Дизайнерские вариации — Claude's Discretion, но контракт «всегда видна, всегда даёт 2 кнопки» — фиксирован.
- **Seed-демо (CD-06)** — один playbook с 5-6 шагами от выдуманного эксперта «Шестаков» (или реального имени, если пользователь подтвердит) критичен для демонстрации walking skeleton. Без него пустой билдер — «ничего не работает». Должен покрывать хотя бы 3 разных action_kind.
- **prerequisites как soft-lock** — UI показывает блокировку, но не запрещает. Это важно: методология может допускать параллельную работу, а жёсткий lock будет бесить пользователя. Реализация — CSS класс `step-blocked` + tooltip «Сначала заверши шаги: A, B».
- **hint-badge (D-18)** — **только** при загрузке страницы вкладки Playbook. Никаких WebSocket'ов, никакого polling'а, никакого auto-refresh. Сменил статус вручную — badge исчезает.
- **Один Markdown-filter на всё** — зарегистрировать `| markdown` в `template_engine.py` один раз. Не писать inline `{{ x | ... }}` трюки.
