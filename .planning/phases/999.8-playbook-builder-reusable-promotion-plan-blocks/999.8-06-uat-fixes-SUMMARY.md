# Phase 999.8 Plan 06: UAT Gap Closure Summary

Фаза 999.8 довела walking-skeleton плейбуков до рабочего состояния — закрыты все 4 независимых блокера, найденные в UAT, плюс каскадом закрылся Gap E (affordance + hint-badges).

## Что было сломано до этого плана

UAT-запуск walking-skeleton (см. `999.8-UAT.md`) выявил 4 independent bug'а, каждый из которых делал следующий шаг сценария невозможным:

| Gap | Симптом | Root cause |
|-----|---------|------------|
| A | `GET /ui/playbooks/blocks?category_id=<uuid>&expert_source_id=` → 422 | FastAPI пытается распарсить пустую строку как `uuid.UUID | None`, валидатор Pydantic отбивает запрос. Пустая строка приходит из sibling `hx-include` на соседнем select-е. |
| B | Клик "Применить" в модале — тишина, модал не закрывается | Endpoint отвечал `HX-Redirect` на тот же URL (`/kanban#playbook`). HTMX воспринимает это как no-op, потому что URL совпадает с текущим (hash уже в history через `switchProjectTab().replaceState`). |
| C | Клик "Перейти к шагу" — silent no-op | Двойной баг: (1) `onclick="openPlaybookStep(..., {{ title \| tojson }}, ...)"` — `tojson` эмитит литеральные `"`, которые коллизят с outer `onclick="..."`; (2) inline `<script>` с определением `window.openPlaybookStep` становится inert после HTMX 2.0 innerHTML swap tab-контента. |
| D | Клик по кружку статуса → 500 MissingGreenlet | `set_step_status()` делал `db.refresh(step)`, что re-expires все relationships. Jinja2 затем читал `step.project_playbook.steps \| length` в синхронном рендере → lazy-load вне greenlet контекста → SQLAlchemy бросает MissingGreenlet. |
| E | "Кружок не кликабельный" + hint-badges не появляются | Каскад от C+D: `opened_at` не штамповался (C был silent), поэтому hint-queries ни за что не цеплялись; курсор не показывал hand (отсутствие `cursor-pointer`). |

## Что сделано — task-by-task

### Task 1 (Gap A) — `app/routers/playbooks.py` :: `ui_playbook_blocks`

**Commit:** `672b3e5`

Сигнатура роута изменена с `uuid.UUID | None` на `str | None` для обоих filter-параметров. В теле функции добавлен локальный `_parse_uuid()` helper, который приводит `""` и невалидные строки к `None` безопасно, без 422. Результаты передаются в `svc.list_blocks()` как `category_uuid` / `expert_uuid`. Контекст шаблона обновлён, чтобы `selected_*` ключи содержали parsed uuids — dropdown по-прежнему подсвечивает выбранные опции.

**НЕ трогал:** `app/templates/playbooks/blocks.html` (`hx-include` pattern корректный), `svc.list_blocks` (уже принимает `None` правильно).

### Task 2 (Gap B) — `app/routers/playbooks.py` :: `ui_project_playbook_apply`

**Commit:** `0c7d504`

Response header заменён с `HX-Redirect: /ui/projects/{id}/kanban#playbook` на `HX-Refresh: true`. `HX-Refresh` заставляет браузер сделать full page reload независимо от текущего URL — модал закрывается, страница перезагружается, URL-hash сохраняется браузером, и существующий IIFE в `kanban.html:177-183` автоматически активирует вкладку Плейбуки. Docstring обновлён под новое поведение.

**НЕ трогал:** `kanban.html` (hash-on-load handler уже существовал), `_apply_modal.html` (не требуется).

### Task 3 (Gap C) — JS extraction + onclick quoting

**Commit:** `3628c26`

Три координированных изменения:

1. **Создан** `app/static/js/playbook_step.js` — IIFE с idempotent guard `if (typeof window.openPlaybookStep === 'function') return;`, содержимое идентично прежнему inline script. Любой authenticated пользователь загружает его один раз при первом открытии страницы, а HTMX innerHTML swap вкладки Плейбуки больше не затрагивает определение функции в `window`.

2. **`app/templates/base.html`** — добавлен `<script src="/static/js/playbook_step.js" defer></script>` сразу после закрывающего `{% endif %}` admin-блока, ВНЕ `{% if is_admin %}`, потому что "Перейти к шагу" должны уметь нажимать все роли.

3. **`app/templates/projects/_playbook_project_step.html`** — три правки:
   - `onclick="..."` → `onclick='...'` с перевёрнутыми вложенными кавычками, чтобы `|tojson` emitting literal `"` корректно лежал внутри double-quoted JSON tokens, а HTML parser смотрел только на внешние `'`.
   - Удалён весь `<script>...</script>` блок (строки 115-162) — логика теперь в static JS.
   - В `class` status-circle кнопки добавлен `cursor-pointer` → hover даёт hand-cursor, закрывает affordance-жалобу из Gap E.

### Task 4 (Gap D) — `app/services/playbook_service.py`

**Commit:** `4b1a7b5`

Три правки в одном файле:

1. **`get_project_step()`** — расширена eager-load chain: вместо `selectinload(ProjectPlaybookStep.project_playbook)` теперь `selectinload(ProjectPlaybookStep.project_playbook).selectinload(ProjectPlaybook.steps)`. Это покрывает чтение `step.project_playbook.steps | length` из шаблона. `ProjectPlaybook` уже был импортирован — новых импортов не потребовалось.

2. **`set_step_status()`** — удалён `await db.refresh(step)`. Session настроена с `expire_on_commit=False` (см. `app/database.py:22-25`), поэтому после `db.flush()` ORM-инстанс сохраняет все eager-loaded relationships. Явный refresh re-expires их и заставляет lazy-load внутри Jinja2 sync context — это и была root cause MissingGreenlet.

3. **`open_step_action()`** — тот же refresh удалён defensive-consistency ради. Функция сейчас возвращает только JSON, поэтому 500 не манифестирует, но если кто-то позже добавит partial render, баг бы вернулся.

**НЕ тронуто:** `add_step()` (строка 536) сохраняет свой `db.refresh(step)` — это Builder flow для fetching server-side defaults на новой `PlaybookStep`, legitimate use, unrelated to Gap D.

## /complete endpoint audit

Плана попросил проверить `api_project_step_complete` (POST `/api/project-playbook-steps/{id}/complete`) — endpoint, который Test 6 вызывает из banner CTA. При проверке обнаружено, что этот endpoint имеет **собственный inline query builder** в `app/routers/playbooks.py:954-961` — он не делегирует в `set_step_status()` и не вызывает `get_project_step()`. Возвращает JSON (`{"status": "ok", "project_id": ...}`), не рендерит partial. Следовательно:

- MissingGreenlet на этом пути **не манифестирует** — нет sync Jinja access.
- Fix в `set_step_status()` не "каскадируется" сюда, но это и **не нужно** — баг здесь и не воспроизводился.
- Speculative refactoring не делал (scope boundary — цель plan'а разблокировать Test 5, а Test 6 уходит через другой путь).

Оставил note для верификатора: если UAT Test 6 всё ещё упадёт — это будет отдельная regression, а не этот баг.
