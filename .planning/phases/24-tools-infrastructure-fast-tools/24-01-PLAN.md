c) Route handlers — register in this order (per RESEARCH Pitfall 5):
  1. `GET /` — tools_index: render tool card grid with job counts per tool
  2. `GET /{slug}/` — tool_landing: render input form + previous jobs list
  3. `POST /{slug}/` — tool_submit: validate input, create Job row, dispatch Celery task, redirect to results page
  4. `GET /{slug}/{job_id}/status` — tool_job_status: HTMX polling partial
  5. `GET /{slug}/{job_id}/export` — tool_export: CSV/XLSX download
  6. `DELETE /{slug}/{job_id}` — tool_delete: delete job + results, return empty 200
  7. `GET /{slug}/{job_id}` — tool_results: results page (LAST — generic catch-all)

For the tools_index route, render with context `{"tools": list(TOOL_REGISTRY.values())}`.

For routes that need models (landing, submit, results, status, export, delete) — add a helper function `_get_tool_models(slug: str)` that lazy-imports the model classes:

```python
def _get_tool_models(slug: str):
    """Lazy-import tool models to avoid circular imports."""
    if slug == "commercialization":
        from app.models.commerce_check_job import CommerceCheckJob, CommerceCheckResult
        return CommerceCheckJob, CommerceCheckResult
    elif slug == "meta-parser":
        from app.models.meta_parse_job import MetaParseJob, MetaParseResult
        return MetaParseJob, MetaParseResult
    elif slug == "relevant-url":
        from app.models.relevant_url_job import RelevantUrlJob, RelevantUrlResult
        return RelevantUrlJob, RelevantUrlResult
    raise HTTPException(status_code=404, detail="Unknown tool")
```

Similarly `_get_tool_task(slug: str)`:
