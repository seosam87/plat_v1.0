# Phase 999.3 Context ‚Äî Smart Route Discovery

## Motivation

Phase 15.1 built `tests/_smoke_helpers.py::discover_routes` which enumerates all GET routes from `app.routes` across 11 UI prefixes. The crawler then asserts status + body markers + structural HTML on every discovered route. During Phase 15.1 execution and the follow-up debug session (`phase-15.1-deferred-routes`), it became clear that the 11-prefix filter over-captures: several routes live under UI prefixes but are not rendered HTML pages ‚Äî they return JSON for Chart.js, CSV downloads, or plain dicts. These had to be individually listed in `SMOKE_SKIP` with per-entry "not a rendered page" reasons. Five such entries were added manually:

```python
"/metrika/{site_id}/pages": "JSON chart-data API under /metrika/ prefix"
"/metrika/{site_id}/compare": "JSON chart-data API under /metrika/ prefix"
"/analytics/sessions/{session_id}/export": "CSV download, not a rendered page"
"/traffic-analysis/sessions/{session_id}": "JSON dict API, not a rendered page"
"/traffic-analysis/sessions/{session_id}/anomalies": "JSON dict API, not a rendered page"
```

This is maintenance debt: every new JSON/CSV endpoint under a UI prefix must be manually discovered + added to SMOKE_SKIP, otherwise the smoke gate fails on ambiguous grounds ("404 because seed doesn't have the model" vs "not a page at all").

**Phase goal:** `discover_routes` filters by response type automatically so that only endpoints FastAPI considers HTML-rendering are subject to the smoke gate. The five manual SMOKE_SKIP entries above are removed. `SMOKE_SKIP` is reserved for its original purpose: explicitly non-testable HTML endpoints (auth redirects, file downloads where the endpoint IS HTML but body-scan is inappropriate, etc.).

## Scope

**In scope:**
- Extend `discover_routes` classification to skip non-HTML response endpoints automatically
- Remove the 5 manual SMOKE_SKIP entries added by phase-15.1-deferred-routes (commit `0e59533`)
- Update `tests/test_smoke_helpers.py` unit tests accordingly
- Run `pytest tests/test_ui_smoke.py` and confirm same pass count (146/146) with 5 fewer SMOKE_SKIP entries
- Document the filter logic inline (docstring) + in `tests/README.md` so future contributors know why their JSON endpoint isn't tested

**Out of scope:**
- Rewriting `discover_routes` from scratch
- Adding query-parameter support (QUERY_MAP) ‚Äî separate concern, stays in backlog
- Adding bind-mount for non-test files (999.5 covers deployment sync)
- Covering POST/DELETE (999.1 Playwright scenario runner covers interactive flows)

## Decisions (locked)

### D-01: Classification signal ‚Äî three-tier priority

A route is considered HTML-rendering if **any** of these hold (checked in order):

1. **Explicit `response_class` on the `APIRoute`**: if `route.response_class` is `HTMLResponse` (or a subclass), include. If it's `JSONResponse`, `PlainTextResponse`, `FileResponse`, `StreamingResponse`, or `Response` ‚Üí **exclude**.
2. **Return-type annotation on the endpoint function**: if `route.endpoint.__annotations__.get("return")` is `HTMLResponse` (or subclass) ‚Üí include. If it's `list`, `dict`, `BaseModel` subclass, `list[dict]`, `dict[str, ...]` ‚Üí **exclude**.
3. **Fallback**: if neither signal is present, **include** (preserve Phase 15.1 behavior ‚Äî conservative default, route goes through the smoke gate).

**Why:** FastAPI's `response_class` is the canonical signal but is often omitted. Return-type annotations are the second-most-reliable signal in this codebase (we use them extensively). The conservative fallback ensures no existing route silently drops off the gate without an explicit signal.

### D-02: Remove exactly the 5 SMOKE_SKIP entries added by commit 0e59533

Those five must be removed and verified by final `pytest tests/test_ui_smoke.py` still green. Do NOT remove any SMOKE_SKIP entry that predates that commit (`/ui/login`, `/ui/logout`, `/ui/logs`, `/`, `/ui/api/sites`, `/ui/content-publish/{site_id}/preview/{job_id}`) ‚Äî those have different rationales (auth flow, root redirect, JSON API already using `/ui/api/` convention, job_id ambiguity).

**Why:** surgical removal, not indiscriminate cleanup. The five target entries are the direct artifact of the manual workaround being replaced.

### D-03: `discover_routes` return shape unchanged

The function currently returns a list of `RouteInfo` (or equivalent) ‚Äî do NOT change the dataclass fields, do NOT rename the function, do NOT change parameters. Only the internal filter logic changes. This keeps blast radius minimal and means `test_ui_smoke.py` needs no modifications.

**Why:** `test_ui_smoke.py` is parametrized over `discover_routes` output ‚Äî any signature change ripples into multiple files and invites regression.

### D-04: Unit-test coverage ‚Äî three new cases in test_smoke_helpers.py

Add unit tests proving:
1. Route with `response_class=HTMLResponse` ‚Üí included
2. Route with `response_class=JSONResponse` ‚Üí excluded
3. Route with no `response_class` but return annotation `-> list[dict]` ‚Üí excluded
4. Route with no `response_class` and no return annotation ‚Üí included (fallback)
5. Route with return annotation `-> HTMLResponse` ‚Üí included

Use in-memory FastAPI test apps (same pattern as existing helper tests), not real `app.routes`. Keeps unit tests fast and isolated.

**Why:** the filter logic is the entire point of the phase; it must be unit-tested independently of the main app.

### D-05: Verification path ‚Äî no regression + SMOKE_SKIP shrinks by exactly 5

Acceptance:
- `pytest tests/test_smoke_helpers.py -x` green (with new cases)
- `pytest tests/test_ui_smoke.py -x` green, same pass count as after phase-15.1-deferred-routes (146/146)
- `grep -c "^\s*\"/" tests/_smoke_helpers.py` in SMOKE_SKIP block: exactly 5 fewer entries than before the phase
- The 5 target routes (metrika pages/compare, analytics export, traffic-analysis sessions √ó 2) do NOT appear as parametrized test IDs in `pytest --collect-only tests/test_ui_smoke.py` output ‚Äî proving the filter worked

**Why:** success isn't "tests pass" ‚Äî it's "tests pass AND the 5 routes are filtered out by the new logic, not by manual skip".

### D-06: Docstring + README update

- `discover_routes` docstring must explain the three-tier classification
- `tests/README.md` section "Adding a new route to smoke coverage" must be updated: if your endpoint returns JSON/CSV/binary, declare `response_class=` OR annotate return type ‚Äî otherwise it will be smoke-tested and you'll need to SMOKE_SKIP it. New contributors should never need to touch SMOKE_SKIP for non-HTML endpoints again.

**Why:** the feature only saves future maintenance if contributors know about it.

### D-07: FastAPI internals ‚Äî handle `response_class` sentinel

FastAPI uses `Default(JSONResponse)` as the default `response_class` when none is declared. The filter must distinguish "user-declared JSONResponse" from "FastAPI default". Check `isinstance(route.response_class, DefaultPlaceholder)` (from `fastapi.datastructures`) ‚Äî if it's a default placeholder, fall through to signal 2 (return annotation) rather than treating it as "user declared JSONResponse".

**Why:** without this, every route without an explicit `response_class` is treated as JSON by signal 1 and falls through to signal 2 ‚Äî which is the intended behavior, but only works if we correctly recognize the placeholder. Getting this wrong silently drops every un-annotated HTML route off the gate.

### D-08: Commit strategy ‚Äî two commits

1. `feat(smoke): filter discover_routes by response_class / return annotation` ‚Äî helper change + unit tests
2. `refactor(smoke): remove 5 manual SMOKE_SKIP entries now handled by filter` ‚Äî SMOKE_SKIP cleanup + final smoke run verification

Split so that if something goes wrong in commit 2, commit 1 stays usable.

## Risks / pitfalls

1. **`DefaultPlaceholder` import path may differ across FastAPI versions** ‚Äî pinned to 0.115, but verify `from fastapi.datastructures import DefaultPlaceholder` works at runtime. If not, use `type(route.response_class).__name__ == "DefaultPlaceholder"` as fallback.
2. **Routes using `response_class=Response`** (generic) may be HTML or not ‚Äî these are rare. Treat as "include" (fallback) since `Response` is ambiguous.
3. **Mounted sub-apps / `include_router` with override** ‚Äî the filter runs on `app.routes` so inherited response_class should propagate correctly, but verify at least one route from each main router passes classification as expected.
4. **A route with `-> list[dict]` annotation that ALSO explicitly returns `HTMLResponse`** (theoretical, unlikely) ‚Äî signal 1 wins, so it would be excluded. Document this edge case in docstring.
