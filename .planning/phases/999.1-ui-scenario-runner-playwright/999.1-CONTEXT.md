# Phase 999.1 Context — UI Scenario Runner (Playwright)

## Goal (from ROADMAP)

YAML-based scenario runner using Playwright async. Steps: open / click / fill / wait_for / expect_text / expect_status. Runs in CI against full docker-compose stack. Reuses seed fixtures from Phase 15.1. Scenarios in `scenarios/*.yaml` — same files later consumed by 999.2 tour player.

## Decisions

### D-01. Runner host: pytest plugin
- Scenarios discovered as pytest test items (`scenarios/*.yaml` collected by a custom pytest collector).
- Reuses existing pytest infra: `tests/fixtures/smoke_seed.py` (`smoke_seed` session fixture, `SMOKE_IDS`), conftest, CI wiring, retry/reporting.
- A standalone CLI is **not** in scope for this phase. If needed later, it should be a thin wrapper that imports the same scenario loader/executor used by the pytest collector.
- **Why:** zero new CI plumbing; downstream 999.2 tour player can still consume the YAML files independently.

### D-02. Stack lifecycle: full docker-compose in CI, attach-mode locally
- **CI:** spin up the full `docker-compose` stack (app + postgres + redis + celery worker), run scenarios against `http://app:8000`. Real browser, real HTMX round-trips, real Celery polling.
- **Local dev:** runner attaches to an already-running `docker compose up` stack via `BASE_URL` env var. No in-process ASGI mode.
- **Why:** the entire purpose of the phase is to exercise interactive flows (HTMX polling, slide-over panels, JS clicks). In-process ASGI cannot reproduce these.

### D-03. Auth: shared session via `storage_state.json`
- Programmatic login executed **once per pytest session** against the seeded admin user.
- Resulting Playwright `storage_state.json` cached and reused by every scenario context.
- A `login:` step type is **reserved** in the YAML schema for future multi-user scenarios but not used by P0 scenarios.
- **Why:** platform is admin-only (single role); per-scenario login adds latency without value.

### D-04. Selectors: role/text first, CSS fallback, no forced data-testid
- **Default:** Playwright role/text locators (`get_by_role`, `get_by_text`, `get_by_label`).
- **Fallback:** raw CSS selectors when role/text is ambiguous (duplicate text, generated ids).
- `data-testid` is supported by the runner but **not** mandated — templates are not modified to sprinkle test ids preemptively.
- YAML step accepts a single `target:` field; runner auto-detects locator type from the value (role-prefixed string vs CSS vs testid).
- **Why:** templates stay untouched, refactor-resilient by default, escape hatch when needed.

### D-05. P0 coverage: 2 scenarios
- **Scenario 1: suggest → results (HTMX polling)** — submit suggest job, poll for completion, assert result rows render. Highest-risk pattern (async Celery + HTMX polling + dynamic DOM).
- **Scenario 2: site form submit** — basic create-site happy path. Validates the runner end-to-end on a simple synchronous form.
- **Deferred to follow-up phases (note in deferred-items):** slide-over detail panels, gap analysis flow, position checks, plus broader HTMX polling coverage.
- **Why:** validate the runner, schema, and CI integration first. Adding more scenarios is cheap once infra works.

### D-06. Failure artifacts: screenshot + Playwright trace.zip
- On scenario failure: capture full-page screenshot **and** Playwright trace (`page.context.tracing.start(snapshots=True, screenshots=True, sources=True)`).
- Both artifacts uploaded by CI as job artifacts under `artifacts/scenarios/{scenario_name}/`.
- Video recording **off by default** — can be enabled per-scenario via YAML or globally via env flag.
- **Why:** `playwright show-trace` gives full replayable diagnostics; video is heavy and rarely needed.

### D-07. YAML schema reserves 999.2 tour step types from day 1
- Step types in P0: `open`, `click`, `fill`, `wait_for`, `expect_text`, `expect_status`.
- **Reserved (not implemented in 999.1, but valid in schema):** `say`, `highlight`, `wait_for_click`.
- Schema validator (JSON Schema or pydantic model) accepts reserved step types but the test runner skips them with a warning (so the same file loads under 999.2 tour player without modification).
- Single source of truth: any `scenarios/*.yaml` must pass validation under both runners.
- **Why:** explicit roadmap goal — "one source of truth for tests and tours". Cheaper to reserve now than to migrate every YAML file later.

## Reused assets (from Phase 15.1)

- `tests/_smoke_helpers.py` — route discovery, response assertions, JINJA_ERROR_MARKERS scanner. Useful as reference for assertion helpers in the new runner.
- `tests/fixtures/smoke_seed.py` — session-scoped DB seed with deterministic `SMOKE_IDS`. The Playwright runner will start from the same seeded state via the same fixture.
- `tests/fixtures/conftest.py` — fixture wiring; Playwright fixtures will live alongside.
- Phase 15.1 SAVEPOINT-per-request pattern is **not** reused — Playwright runs against the live stack, not in-process ASGI, so each scenario sees a real DB.

## Open questions for research

- Best 2026 pattern for Playwright async + pytest-asyncio session-scoped browser context (avoid one-browser-per-test overhead).
- How to wait deterministically for HTMX swap completion (custom event listener vs `wait_for` on result selector vs `htmx:afterSwap` script-injection).
- CI image: extend existing `mcr.microsoft.com/playwright/python:v1.47.0-jammy` or install Playwright into the existing app image — size vs cache trade-off.
- YAML schema design: pydantic model vs JSON Schema vs both.

## Deferred ideas (not in this phase)

- Standalone scenario runner CLI (`python -m scenario_runner`) — only if 999.2 tour player needs it and the pytest entry point doesn't fit.
- Slide-over panel scenario, gap analysis scenario, position-check scenario — backlog for follow-up phase.
- Per-scenario video recording — flag exists, default off.
- `data-testid` migration across all templates — only add on a case-by-case basis when role/text locators genuinely fail.
- Multi-user scenarios using the reserved `login:` step type.

## Constraints inherited from PROJECT.md

- Python 3.12, async-first, FastAPI 0.115, HTMX 2.0
- Playwright 1.47+ via official Microsoft image where possible
- No new tables, no migrations (runner is test-only infra)
- Admin-only platform — single user role
