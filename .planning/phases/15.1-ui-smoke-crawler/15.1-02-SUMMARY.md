# Phase 15.1 Plan 02: Smoke Crawler Helpers Summary

**One-liner:** Pure, unit-tested helper module (`tests/_smoke_helpers.py`) providing route discovery from `app.routes`, path-param resolution, and response-body assertions — the foundation that Plan 03 will wire into the parametrized smoke test.

## What shipped

- `tests/_smoke_helpers.py` (~165 lines): `discover_routes()`, `build_param_map()`, `resolve_path()`, `assert_no_error_markers()`, `assert_structural_html()`, `is_partial()`, plus `UI_PREFIXES` (11 entries), `SMOKE_SKIP` dict with reason comments, `JINJA_ERROR_MARKERS`, `RouteSpec` dataclass, `UnknownParamError`.
- `tests/test_smoke_helpers.py`: 42 unit tests covering every exported symbol, every UI prefix (parametrized), every Jinja error marker (parametrized), the `builtin_function_or_method` regression, Starlette typed-converter paths, and every branch of `assert_structural_html` and `is_partial`.

## SMOKE_SKIP — discovered paths

Grep of `app/routers/` and `app/main.py` for `{job_id}` routes yielded:

| Path | Router | Reason skipped |
|---|---|---|
| `/` | `app/main.py` root handler | Redirect to `/ui/dashboard`, not a template render |
| `/ui/login` | auth router | Auth entry point — redirect loop under test client |
| `/ui/logout` | auth router | 302 session destruction |
| `/ui/logs` | admin router | Live tail endpoint |
| `/ui/api/sites` | sites router | JSON-only fragment |
| `/ui/content-publish/{site_id}/preview/{job_id}` | `app/main.py:1221` | `job_id` collides between `SuggestJob` and `WpContentJob` — single PARAM_MAP cannot satisfy both. Deferred to backlog 999.1. |

**Note:** `wp_pipeline` router (`app/routers/wp_pipeline.py`) registers under prefix `/pipeline/`, which is NOT in `UI_PREFIXES`, so its `GET /pipeline/jobs/{job_id}` is filtered out automatically by the prefix gate — no explicit `SMOKE_SKIP` entry required.

## Deviations from Plan

**None of Rules 1-3 triggered.** Plan executed exactly as written. One clarification: the plan suggested grepping for wp_pipeline paths under UI prefixes; the actual wp_pipeline router lives at `/pipeline/` which is already filtered by `UI_PREFIXES`, so only the content-publish preview route needed an explicit skip entry. This is documented in the decisions above.
