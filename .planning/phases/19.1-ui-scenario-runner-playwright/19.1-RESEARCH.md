# Phase 19.1: UI Scenario Runner (Playwright) — Research

**Researched:** 2026-04-08
**Domain:** Browser test runner / Playwright async / pytest custom collector / HTMX synchronization
**Confidence:** HIGH (decisions locked in CONTEXT; research focuses on implementation patterns)

## Summary

Phase 19.1 builds a YAML-driven Playwright scenario runner that plugs into the existing pytest infrastructure as a **custom file collector** and runs against the **live docker-compose stack** in CI. Two P0 scenarios validate the runner end-to-end: a synchronous form submit and an async HTMX-polling flow (suggest → results).

The biggest implementation pivot vs Phase 15.1 is that **`smoke_seed` cannot be reused as-is**. Phase 15.1's SAVEPOINT-bound seed runs in-process against the test ASGI app — Playwright runs out-of-process against a real HTTP server inside Docker. We need a separate seed strategy for the live stack (recommendation: a Python seed module reusing `SMOKE_IDS` constants, executed inside the `api` container at CI startup, idempotent, committing for real). `SMOKE_IDS` itself is reusable verbatim — that's the value of keeping the constants centralized.

**Primary recommendation:** Custom pytest collector (`pytest_collect_file`) yielding `pytest.Item`s per scenario, raw `playwright.async_api` (NOT pytest-playwright — incompatible with custom collectors), one session-scoped Chromium + per-scenario `BrowserContext` constructed from a cached `storage_state.json`. HTMX waits use `expect(locator).to_be_visible()` on the swapped target element (idiomatic Playwright auto-wait, deterministic, no `htmx:afterSwap` script injection needed). YAML schema validated by a Pydantic v2 discriminated union over a `Step` base. Failure hook stops the trace, saves `trace.zip` + screenshot under `artifacts/scenarios/{scenario}/`. Live-stack seed via a one-shot `python -m tests.scenario_runner.seed` invocation inside the `api` container after `alembic upgrade head`.

### Pattern 1: Custom pytest collector for YAML
**What:** A `pytest_collect_file` hook that returns a `pytest.File` subclass whose `collect()` yields one `pytest.Item` per scenario file. The Item's `runtest()` calls the async executor.

**When to use:** Phase 19.1 only — this is the canonical pattern for "data file as test."

**Example (skeleton — pytest 8.x API):**
```python
# tests/fixtures/scenario_runner/collector.py
import pytest, yaml, asyncio
from pathlib import Path
from .schema import Scenario
from .executor import run_scenario

def pytest_collect_file(parent, file_path: Path):
    if file_path.suffix == ".yaml" and file_path.parent.name == "scenarios":
        return ScenarioFile.from_parent(parent, path=file_path)

class ScenarioFile(pytest.File):
    def collect(self):
        raw = yaml.safe_load(self.path.read_text())
        scenario = Scenario.model_validate(raw)
        yield ScenarioItem.from_parent(self, name=scenario.name, scenario=scenario)

class ScenarioItem(pytest.Item):
    def __init__(self, *, scenario, **kw):
        super().__init__(**kw)
        self.scenario = scenario

    def runtest(self):
        # Pull session-scoped fixtures via self._request
        page = self._request.getfixturevalue("scenario_page")
        asyncio.get_event_loop().run_until_complete(run_scenario(page, self.scenario))

    def repr_failure(self, excinfo):
        return f"Scenario {self.name} failed:\n{excinfo.getrepr()}"

    def reportinfo(self):
        return self.path, 0, f"scenario: {self.name}"
```

**Sharing fixtures with collected items:** `pytest.Item` does not get fixtures injected automatically (fixtures bind to test functions). Two patterns work:
1. **Recommended:** Override `setup()` on the Item to call `self._request.getfixturevalue("scenario_page")` — gives full session-scoped fixture access (`smoke_seed_ids`, `scenario_browser`, `scenario_page`).
2. Use a `@pytest.fixture(autouse=True)` that pokes data onto the Item — clunkier.

The `_request` attribute is set by pytest during collection finalization; if it's not available, use `self.session._fixturemanager.getfixtureclosure(...)` — but `getfixturevalue` from a wrapping function fixture is the documented path.

**Source:** pytest 8 docs — Working with non-python tests / `pytest_collect_file`. https://docs.pytest.org/en/8.x/example/nonpython.html

### Pattern 2: Session-scoped Playwright browser, per-scenario context
**What:** One Chromium per pytest session (5–10s startup amortized), fresh `BrowserContext` per scenario (cheap, ~50ms), each context loads cached `storage_state.json` for auth.

**When to use:** Always for this phase. Avoids per-scenario browser launch overhead while keeping scenarios isolated (cookies, localStorage don't bleed).

**Example:**
```python
# tests/fixtures/scenario_runner/conftest.py
import pytest_asyncio, json
from pathlib import Path
from playwright.async_api import async_playwright

STORAGE_STATE = Path("artifacts/.auth/storage_state.json")

@pytest_asyncio.fixture(scope="session")
async def scenario_playwright():
    async with async_playwright() as pw:
        yield pw

@pytest_asyncio.fixture(scope="session")
async def scenario_browser(scenario_playwright):
    browser = await scenario_playwright.chromium.launch(headless=True)
    yield browser
    await browser.close()

@pytest_asyncio.fixture(scope="session")
async def storage_state(scenario_browser, base_url):
    if STORAGE_STATE.exists():
        return str(STORAGE_STATE)
    ctx = await scenario_browser.new_context()
    page = await ctx.new_page()
    await page.goto(f"{base_url}/ui/login")
    await page.get_by_label("Username").fill("smoke_admin")
    await page.get_by_label("Password").fill("smoke-password")
    await page.get_by_role("button", name="Log in").click()
    await page.wait_for_url(f"{base_url}/ui/dashboard")
    STORAGE_STATE.parent.mkdir(parents=True, exist_ok=True)
    await ctx.storage_state(path=str(STORAGE_STATE))
    await ctx.close()
    return str(STORAGE_STATE)

@pytest_asyncio.fixture
async def scenario_page(scenario_browser, storage_state):
    ctx = await scenario_browser.new_context(storage_state=storage_state)
    await ctx.tracing.start(snapshots=True, screenshots=True, sources=True)
    page = await ctx.new_page()
    yield page
    await ctx.close()
```

**Why NOT pytest-playwright:** pytest-playwright provides `page` / `browser` fixtures bound to test *functions*. Custom collected `pytest.Item`s don't get those fixtures automatically (no function signature to inspect), and pytest-playwright's plugin hooks into `pytest_runtest_setup` in ways that conflict with `getfixturevalue` from `pytest.Item.setup()`. The raw `playwright.async_api` is ~30 lines and avoids the conflict entirely.

### Pattern 3: HTMX swap synchronization
**What:** Use Playwright's auto-waiting `expect(locator).to_be_visible(timeout=N)` against the **swapped target element** — not the trigger button, not document state.

**When to use:** Every HTMX wait. This is the idiomatic Playwright way; auto-retry handles polling-based swaps (suggest → results) naturally.

**Why this beats `htmx:afterSwap` script injection:** The event-listener approach has a race window — if the swap completes before the listener attaches, the test hangs. Auto-waiting locators have no race because they retry on a polling loop until the DOM matches.

**Example (suggest → results scenario, the hard one):**
```python
# In executor.py for `wait_for: { selector: "#results-table tr.result-row", state: visible, timeout: 60000 }`
from playwright.async_api import expect

async def step_wait_for(page, target: str, state: str, timeout_ms: int):
    locator = resolve_locator(page, target)
    if state == "visible":
        await expect(locator).to_be_visible(timeout=timeout_ms)
    elif state == "attached":
        await expect(locator).to_be_attached(timeout=timeout_ms)
    elif state == "hidden":
        await expect(locator).to_be_hidden(timeout=timeout_ms)
```

For the **suggest → results scenario specifically**: HTMX polls (`hx-trigger="every 2s"`) the job-status endpoint until `status=complete`, then swaps the results table in. Wait pattern:
```yaml
- click: { target: "role=button[name='Run suggest']" }
- wait_for: { target: "#suggest-results tr.result-row >> nth=0", state: "visible", timeout: 60000 }
- expect_text: { target: "#suggest-results", contains: "smoke seed" }
```
The 60s timeout covers Celery + Wordstat round-trip in CI. Locator auto-retries on the polled DOM — zero scripting needed.

**Escape hatch (only if auto-wait misbehaves):** Inject a Promise that resolves on `htmx:afterSwap`:
```python
await page.evaluate("""
() => new Promise(r => document.body.addEventListener('htmx:afterSwap', r, {once:true}))
""")
```
Document this as a last resort; do not use in P0 scenarios.

**Source:** Playwright auto-waiting docs (https://playwright.dev/python/docs/actionability) and `expect(locator)` assertions (https://playwright.dev/python/docs/test-assertions).

### Pattern 4: Pydantic v2 discriminated union for step schema
```python
# tests/fixtures/scenario_runner/schema.py
from typing import Literal, Annotated, Union
from pydantic import BaseModel, Field

class OpenStep(BaseModel):
    op: Literal["open"]
    url: str

class ClickStep(BaseModel):
    op: Literal["click"]
    target: str

class FillStep(BaseModel):
    op: Literal["fill"]
    target: str
    value: str

class WaitForStep(BaseModel):
    op: Literal["wait_for"]
    target: str
    state: Literal["visible", "hidden", "attached"] = "visible"
    timeout: int = 30000

class ExpectTextStep(BaseModel):
    op: Literal["expect_text"]
    target: str
    contains: str

class ExpectStatusStep(BaseModel):
    op: Literal["expect_status"]
    code: int

# Reserved for 19.2 tour player — schema-valid, runner skips with warning.
class SayStep(BaseModel):
    op: Literal["say"]
    text: str

class HighlightStep(BaseModel):
    op: Literal["highlight"]
    target: str

class WaitForClickStep(BaseModel):
    op: Literal["wait_for_click"]
    target: str

Step = Annotated[
    Union[OpenStep, ClickStep, FillStep, WaitForStep, ExpectTextStep,
          ExpectStatusStep, SayStep, HighlightStep, WaitForClickStep],
    Field(discriminator="op"),
]

class Scenario(BaseModel):
    name: str
    description: str | None = None
    steps: list[Step]
```

**Why discriminated union vs JSON Schema:** Pydantic v2 discriminated unions give O(1) dispatch on the `op` field, produce error messages that point at the offending step (`steps.3.op: 'clik' does not match any of...`), and integrate with mypy. JSON Schema would need a separate validator step and worse error reporting. Pydantic is already required by the project — no new dep.

**Source:** Pydantic v2 docs — Discriminated Unions. https://docs.pydantic.dev/latest/concepts/unions/#discriminated-unions

### Anti-Patterns to Avoid
- **`page.wait_for_timeout(N)` for HTMX waits** — flaky; use auto-waiting locators.
- **One browser per scenario** — 5–10s startup overhead per scenario kills the runner.
- **Per-scenario login** — D-03 violation; reuse storage_state.
- **`page.evaluate` for assertions** — bypasses Playwright's retry logic; use `expect(locator)`.
- **Pydantic v1 syntax (`@validator`, `Config` class)** — CLAUDE.md forbids; use v2 (`@field_validator`, `model_config`).
- **Reusing in-process `smoke_seed` against the live stack** — won't work, see Topic 8.

## Common Pitfalls

### Pitfall 1: pytest-playwright vs custom collectors
**What goes wrong:** Engineer installs `pytest-playwright` for the `page` fixture, then can't figure out why collected `ScenarioItem`s can't access it.
**Why:** pytest-playwright fixtures are function-scoped and bind to test function signatures via fixture closure; collected Items have no signature.
**Avoid:** Don't install pytest-playwright. Use raw `playwright.async_api` in a session-scoped pytest_asyncio fixture (Pattern 2).
**Warning sign:** `fixture 'page' not found` errors from collector setup.

### Pitfall 2: Event loop scope mismatch with smoke_seed
**What goes wrong:** `smoke_seed` is session-scoped under `pytest-asyncio` — if the scenario runner's `scenario_browser` uses a different loop scope, you get `RuntimeError: Event loop is closed` or `attached to a different loop`.
**Why:** pytest-asyncio creates one event loop per scope.
**Avoid:** Pin both `smoke_seed` (if reused for `SMOKE_IDS` only — fine) and `scenario_browser` to `scope="session"`. Better: don't reuse `smoke_seed` at all in this phase — only import `SMOKE_IDS` constants from it.
**Warning sign:** Tests pass alone, fail when run alongside Phase 15.1 smoke tests.

### Pitfall 3: HTMX swap arrives before locator query starts
**What goes wrong:** Test calls `page.locator(...).wait_for()` *after* the swap already completed and HTMX has moved on. Some implementations of `wait_for_selector` with `state=attached` succeed regardless, but custom event listeners hang forever.
**Why:** Race between client polling and test polling.
**Avoid:** Use `expect(locator).to_be_visible(timeout=N)` — it polls the DOM continuously until match, no race.
**Warning sign:** Test passes locally, hangs in CI; or vice versa.

### Pitfall 4: storage_state.json captured against wrong base_url
**What goes wrong:** Cached `storage_state.json` from local dev (cookies for `localhost:8000`) is checked into git or persisted between CI runs and is replayed against `http://app:8000`. Cookies don't apply (different host) → silent re-redirect to /ui/login → first scenario fails on a "fill #username" step that doesn't exist on the post-login page.
**Avoid:** Generate `storage_state.json` fresh every CI run (write to `artifacts/.auth/`, gitignored, no caching across runs). Make the `storage_state` fixture session-scoped but **not** persisted across sessions in CI.
**Warning sign:** First scenario fails; later scenarios pass.

### Pitfall 5: Celery worker not ready when scenario runs
**What goes wrong:** `docker compose up -d --wait` returns when healthchecks pass, but the `worker` service has no healthcheck — it might be mid-import when the suggest scenario fires its first job. Job sits in queue forever; scenario times out.
**Avoid:** Add a `healthcheck` to the `worker` service in `docker-compose.yml` (or a separate `docker-compose.ci.yml` overlay): `celery -A app.celery_app inspect ping -d celery@$$HOSTNAME`. Or poll `flower` API in a CI step before running pytest.
**Warning sign:** suggest → results scenario times out at 60s, manual rerun succeeds.

### Pitfall 6: Reserved step types fail validation
**What goes wrong:** `say` / `highlight` / `wait_for_click` are in the discriminated union but the executor's match statement raises `KeyError`.
**Avoid:** Executor's step dispatch must explicitly skip reserved types with a warning log: `logger.warning(f"step '{step.op}' reserved for 19.2 tour player, skipping")`. Test this — add a third "reserved-types-skipped" YAML to the suite (not a P0 scenario, but a regression check).
**Warning sign:** A 19.2 tour file fails when run under 19.1 — D-07 violated.

## Code Examples

### Locator auto-detection
```python
# tests/fixtures/scenario_runner/locators.py
def resolve_locator(page, target: str):
    if target.startswith("role="):
        # role=button[name="Submit"]
        import re
        m = re.match(r'role=(\w+)(?:\[name=["\'](.+?)["\']\])?', target)
        kw = {"name": m.group(2)} if m.group(2) else {}
        return page.get_by_role(m.group(1), **kw)
    if target.startswith("text="):
        return page.get_by_text(target[5:])
    if target.startswith("label="):
        return page.get_by_label(target[6:])
    if target.startswith("testid="):
        return page.get_by_test_id(target[7:])
    return page.locator(target)  # CSS fallback
```

### Failure hook (artifacts)
```python
# In ScenarioItem.runtest() — wrap in try/except:
async def _run(self):
    page = self._request.getfixturevalue("scenario_page")
    ctx = page.context
    try:
        await run_scenario(page, self.scenario)
    except Exception:
        out_dir = Path(f"artifacts/scenarios/{self.scenario.name}")
        out_dir.mkdir(parents=True, exist_ok=True)
        await page.screenshot(path=str(out_dir / "failure.png"), full_page=True)
        await ctx.tracing.stop(path=str(out_dir / "trace.zip"))
        raise
    else:
        await ctx.tracing.stop()  # discard trace on success
```

### Out-of-process seed (live stack)
```python
# tests/fixtures/scenario_runner/seed.py — runnable as `python -m tests.fixtures.scenario_runner.seed`
"""Idempotent live-stack seed. Reuses SMOKE_IDS so YAML scenarios reference
the same UUIDs as Phase 15.1 in-process tests.

Differences from smoke_seed.py:
  - Commits for real (no SAVEPOINT rollback)
  - Idempotent: ON CONFLICT DO NOTHING via merge() or pre-check
  - Runs against the production engine inside the api container
  - Does NOT touch app.dependency_overrides (out of process)
"""
import asyncio
from sqlalchemy.ext.asyncio import async_sessionmaker, create_async_engine
from app.config import settings
from app.database import Base
from tests.fixtures.smoke_seed import SMOKE_IDS, _seed_core, _seed_extended

async def main():
    engine = create_async_engine(settings.DATABASE_URL, echo=False)
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all, checkfirst=True)
    Session = async_sessionmaker(engine, expire_on_commit=False)
    async with Session() as s:
        # Idempotency: check if smoke_admin already exists
        from app.models.user import User
        from uuid import UUID
        existing = await s.get(User, UUID(SMOKE_IDS["user_id"]))
        if existing is None:
            await _seed_core(s)
            await _seed_extended(s)
            await s.commit()
            print(f"Seeded {len(SMOKE_IDS)} entities")
        else:
            print("Already seeded, skipping")
    await engine.dispose()

if __name__ == "__main__":
    asyncio.run(main())
```

**Important caveat:** `_seed_core` and `_seed_extended` in `smoke_seed.py` are currently `async def _seed_core(session: AsyncSession)` — they're reusable as-is *if* they don't depend on the SAVEPOINT-bound session behavior. They use `session.flush()` only (no commit), which is correct — the outer caller decides whether to commit (live seed) or rollback (smoke test). This means **`smoke_seed.py` needs a small refactor**: extract `_seed_core` / `_seed_extended` so they can be imported by `scenario_runner/seed.py` without dragging in the SAVEPOINT machinery. The functions are already pure-on-session, so the refactor is just making them module-level public (`seed_core` / `seed_extended`) — no logic changes.

### CI image strategy recommendation

**Recommendation:** Add a separate `tester` service in a `docker-compose.ci.yml` overlay using `mcr.microsoft.com/playwright/python:v1.47.0-jammy` as the base. Mount the repo, install only project deps (Playwright is pre-installed in the image), run pytest from inside the container so it can reach `api:8000` over the compose network.

**Why not extend the existing `api` image:** The Playwright base image is ~1.5 GB. The `api` image is ~300 MB. Extending it would inflate every prod build for a test-only need.

**Why not install Playwright into the existing app image at CI time:** Slow (Chromium download every CI run if not cached), brittle (system deps for Chromium are not in `python:3.12-slim`), and pollutes the image.

**Trade-offs of separate tester service:**
- ✓ Fully cached after first CI pull (image cache)
- ✓ Zero impact on prod image
- ✓ Reaches `api` over compose network with no port forwarding
- ✗ One extra image to pull (~1.5 GB first time, then cached)
- ✗ Tests can't import app code unless the repo is volume-mounted (it will be — mount `./app:/app/app` and `./tests:/app/tests`)

```yaml
# docker-compose.ci.yml (overlay)
services:
  tester:
    image: mcr.microsoft.com/playwright/python:v1.47.0-jammy
    working_dir: /app
    volumes:
      - .:/app
    environment:
      BASE_URL: http://api:8000
      DATABASE_URL: ${DATABASE_URL}
    depends_on:
      api:
        condition: service_healthy
    networks:
      - default
    command: >
      sh -c "pip install -r requirements-test.txt &&
             pytest scenarios/ -v"
```

CI invocation: `docker compose -f docker-compose.yml -f docker-compose.ci.yml up --abort-on-container-exit --exit-code-from tester tester`

## Live-stack DB seed — option analysis (Topic 8 deep-dive)

Three options were considered:

| Option | Pros | Cons | Verdict |
|---|---|---|---|
| (a) Python seed module run inside `api` container at CI startup | Reuses `seed_core` / `seed_extended` verbatim; idempotent; no app code change; runs in same env as production | Requires `docker compose exec` step in CI workflow | **Recommended** |
| (b) Alembic data migration loaded only in test profile | Lives with schema; runs automatically on `alembic upgrade head` | Pollutes production migration history; needs profile gating logic that doesn't exist | Rejected — pollutes prod migrations |
| (c) FastAPI dev-only seed endpoint gated by env var | Cleanest CI step (single curl) | Adds production code surface for test-only need; security concern even if env-gated; CLAUDE.md "no test code in app" spirit | Rejected — adds prod surface |

**Recommendation:** Option (a). The seed module lives in `tests/fixtures/scenario_runner/seed.py`, imports `seed_core` / `seed_extended` from `tests/fixtures/smoke_seed.py` (after the underscore-rename refactor), commits for real, and is idempotent (checks for `smoke_admin` user before inserting). CI invokes it via `docker compose exec -T api python -m tests.fixtures.scenario_runner.seed`. Local dev runs the same command against an attached stack.

The `tests/` directory is already mounted into the `api` container (per `docker-compose.yml` line 45), so no new mount is needed.

## Key Findings (executive summary for orchestrator)

1. **`smoke_seed` cannot be reused for live stack** — needs an out-of-process idempotent seed module that imports `seed_core` / `seed_extended` after a one-line underscore-rename refactor in `smoke_seed.py`.
2. **Do NOT install pytest-playwright** — incompatible with custom collectors. Use raw `playwright.async_api` in session-scoped fixtures.
3. **HTMX waits = `expect(locator).to_be_visible(timeout=N)`** — no `htmx:afterSwap` script injection needed for P0.
4. **Pydantic v2 discriminated union on `op` field** is the right schema tech; no JSON Schema needed.
5. **Separate `tester` service** in a `docker-compose.ci.yml` overlay using `mcr.microsoft.com/playwright/python:v1.47.0-jammy`; do NOT extend the prod `api` image with Playwright.
6. **Worker healthcheck is missing** from `docker-compose.yml` and will cause flaky CI runs on the suggest scenario — Wave 0 must add it (in the CI overlay, not prod compose).
7. **Reserved step types (`say` / `highlight` / `wait_for_click`)** must be in the schema from day 1 AND the executor must explicitly skip-with-warning, or 19.2 tour YAMLs will fail under 19.1.
