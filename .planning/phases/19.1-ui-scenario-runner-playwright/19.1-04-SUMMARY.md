## Accomplishments

- **Compose overlay** (`/opt/seo-platform/docker-compose.ci.yml`): no `version:` field (modern compose). Three service entries:
  - `api` — adds a healthcheck (`curl -fsS http://localhost:8000/health`). Base compose has no api healthcheck; without one, `depends_on: api: service_healthy` on the tester would fail to resolve.
  - `worker` — adds the Celery inspect-ping healthcheck with doubled `$$HOSTNAME` so compose parses to `$HOSTNAME` and the container shell expands it at exec time against the container's own hostname.
  - `tester` — new service using `mcr.microsoft.com/playwright/python:v1.47.0-jammy`, mounts the repo at `/app`, sets `BASE_URL=http://api:8000` over the compose network, depends_on api + worker `service_healthy`, and runs `pip install --no-cache-dir -e .` followed by `pytest scenarios/ tests/fixtures/scenario_runner/ -v --tb=short`.
- **Merged config validates:** `docker compose -f docker-compose.yml -f docker-compose.ci.yml config --quiet` returns 0, and `config | grep playwright/python` confirms the tester image is present in the merged graph.
- **CI entrypoint** (`/opt/seo-platform/scripts/run-scenarios-ci.sh`, mode 755):
  1. `cd` to repo root via `$(dirname "$0")/..`
  2. `docker compose ... up -d --wait postgres redis api worker beat` — blocks until all healthchecks report healthy.
  3. `docker compose ... exec -T api python -m tests.fixtures.scenario_runner.seed` — idempotent seed from plan 03.
  4. `docker compose ... up --abort-on-container-exit --exit-code-from tester tester` — runs the suite and bubbles pytest's exit code.
  5. On non-zero exit, lists `artifacts/scenarios/` so failure artifacts are visible in CI logs.
  6. `exit $TESTER_EXIT` — script's exit code mirrors pytest.
- **Script passes `bash -n` syntax check** and is executable.

## Decisions Made

- **api healthcheck lives in the overlay, not the base file** — the plan explicitly allows adding it if the base file lacks one (it does), and keeping it in the overlay preserves the "prod compose untouched, test concerns in overlay" separation this plan exists to enforce.
- **`$$HOSTNAME` escape in the worker healthcheck** — compose YAML interpolates single `$`. A literal `$HOSTNAME` would be resolved at compose-parse time against the host machine's shell env, pinning the healthcheck to an arbitrary string. `$$HOSTNAME` escapes to `$HOSTNAME` in the generated container command, where the in-container shell expands it to the container's actual hostname — which is what `celery -d celery@<hostname>` expects.
- **`set +e` guard around the tester run** — the script runs under `set -euo pipefail` for safety. Pytest failures are *expected* outcomes we want to report, not script bugs. Toggling `set -e` off for exactly one line and capturing `$?` is cleaner than reworking the whole script to trap ERR.
- **No GitHub Actions workflow changes** — a `.github/workflows/test.yml` already exists, but per the plan this is a follow-up. The shell entrypoint is the portable surface; any CI can invoke it.
