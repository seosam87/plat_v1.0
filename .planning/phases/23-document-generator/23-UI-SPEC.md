### 7. Confirm Dialog Pattern

Uses native `confirm()` — consistent with existing delete/destructive patterns project-wide:

```javascript
function confirmSend(docId, channel, address) {
  var msg = channel === 'email'
    ? 'Отправить документ на ' + address + ' через Email?'
    : 'Отправить документ в Telegram?';
  if (!confirm(msg)) return;
  htmx.ajax('POST', '/ui/crm/clients/.../documents/' + docId + '/send',
    {values: {channel: channel}, target: '#toast-container', swap: 'beforeend'});
}

function confirmRegenerate(docId) {
  if (!confirm('Перегенерировать документ? Будет создана новая версия.')) return;
  htmx.ajax('POST', '/ui/crm/clients/.../documents/' + docId + '/regenerate',
    {target: '#documents-list', swap: 'afterbegin'});
}
```

## Interaction Contract

### Flow: Generate Document

1. User is on `/ui/crm/clients/{id}` → clicks "Документы" tab
2. Tab panel shows generate form (top) + empty state or document table (bottom)
3. User selects template from dropdown, selects site (required), clicks "Создать документ"
4. HTMX POST → server creates GeneratedDocument record (status=pending) + dispatches Celery task
5. Server returns `_gen_status.html` partial (generating state) → inserted `afterbegin` in `#documents-list`
6. HTMX polls `/ui/crm/clients/{id}/documents/{doc_id}/status` every 3 seconds
7. When status=ready: server returns terminal partial + `HX-Trigger: refreshDocList` header
8. Client refreshes `#documents-list` → new row appears with status badge "Готов"

### Flow: Download

1. User clicks "Скачать" button (only visible when status=ready)
2. Browser navigates to `/ui/crm/clients/{id}/documents/{doc_id}/download`
3. Server responds with `Content-Disposition: attachment; filename="{file_name}"`
4. Browser download dialog appears — no HTMX involved

### Flow: Send

1. User clicks "Отправить ▼" → dropdown appears
2. User clicks Email or Telegram option
3. `confirmSend()` runs native `confirm()` dialog
4. If OK: HTMX POST → server dispatches Celery send task
5. Server returns toast partial → appended to `#toast-container`
6. Toast shows: "Документ отправлен" (success) or "Ошибка отправки" (failure)

### Flow: Regenerate

1. User clicks "Перегенерировать" → `confirmRegenerate()` runs native `confirm()` dialog
2. If OK: HTMX POST → server enforces 3-version cap, creates new GeneratedDocument (status=pending), dispatches Celery task
3. Server returns `_gen_status.html` partial (generating state) → inserted `afterbegin` in `#documents-list`
4. Polls to completion same as generate flow

### Flow: Delete

1. User clicks "Удалить" → native `confirm()` via `onsubmit` handler
2. Form POST → hard delete → server responds with HX-Trigger or redirect
3. Row removed from table

## Destructive Actions

| Action | Trigger | Confirmation Approach | Recovery |
|--------|---------|----------------------|---------|
| Delete document | "Удалить" button | `confirm()` in `onsubmit` on form | None — hard delete |
| Regenerate (overwrites oldest version when at cap) | "Перегенерировать" | `confirmRegenerate()` with `confirm()` | Prior version deleted; user informed |

Note: "Regenerate" is not destructive in the typical sense (it creates a new document), but when the 3-version cap is enforced, the oldest version is silently deleted. The confirm dialog copy ("Будет создана новая версия") implicitly informs the user without alarming them unnecessarily.
