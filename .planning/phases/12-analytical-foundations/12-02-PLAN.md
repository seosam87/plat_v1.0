<objective>
Build the Quick Wins page — a sortable, filterable table showing pages with positions 4-20 that have unfixed SEO issues, ranked by opportunity score. Users can select pages and dispatch batch fixes (TOC/Schema/Links) through the existing content pipeline.

Purpose: Quick Wins surfaces the highest-ROI pages — close to page 1 but held back by fixable issues. The opportunity score formula (21-position) x weekly_traffic ensures the most impactful pages appear first.

Output: Quick Wins service, router, template with batch fix modal, navigation update, service tests
</objective>

    **1. `async def get_quick_wins(db: AsyncSession, site_id: uuid.UUID, issue_type: str | None = None, content_type: str | None = None) -> list[dict]`**

    SQL logic (raw SQL via `text()` for complex JOINs):
    ```sql
    WITH page_positions AS (
        SELECT
            klp.url AS page_url,
            AVG(klp.position) AS avg_position,
            COUNT(klp.keyword_id) AS keyword_count
        FROM keyword_latest_positions klp
        WHERE klp.site_id = :site_id
          AND klp.position BETWEEN 4 AND 20
        GROUP BY klp.url
        HAVING AVG(klp.position) BETWEEN 4 AND 20
    ),
    page_traffic AS (
        SELECT page_url, SUM(visits) AS weekly_traffic
        FROM metrika_traffic_pages
        WHERE site_id = :site_id
          AND period_end >= CURRENT_DATE - INTERVAL '7 days'
        GROUP BY page_url
    )
    SELECT
        p.id AS page_id, p.url, p.content_type, p.has_toc, p.has_schema,
        p.word_count, p.inlinks_count,
        pp.avg_position,
        COALESCE(pt.weekly_traffic, 0) AS weekly_traffic,
        (21 - pp.avg_position) * COALESCE(pt.weekly_traffic, 0) AS opportunity_score
    FROM page_positions pp
    JOIN pages p ON normalize_url(p.url) = normalize_url(pp.page_url) AND p.site_id = :site_id
    LEFT JOIN page_traffic pt ON normalize_url(pt.page_url) = normalize_url(pp.page_url)
    ```

    NOTE: Since normalize_url is Python-side, the JOIN must use a subquery approach or Python-side normalization. Preferred approach: use Python normalization when building the result set. Execute the CTE to get page_urls, normalize them in Python, then look up pages and traffic.

    Actual approach:
    1. Query keyword_latest_positions grouped by url WHERE position BETWEEN 4 AND 20
    2. For each url, normalize it in Python
    3. Query pages table matching normalized URLs for the site (get latest crawl pages)
    4. Query metrika_traffic_pages for weekly traffic
    5. Compute opportunity_score = (21 - avg_position) * weekly_traffic
    6. Filter: only pages where at least one audit check is "fail" (has unfixed issue)
    7. Apply issue_type filter if provided: "missing_toc" (has_toc=False), "missing_schema" (has_schema=False), "low_links" (inlinks_count < 3), "thin_content" (word_count < 300)
    8. Apply content_type filter if provided
    9. Sort by opportunity_score DESC (per D-04)
    10. Return list of dicts with keys: page_id, url, opportunity_score, has_toc, has_schema, has_low_links (bool), has_thin_content (bool), avg_position (rounded to 1 decimal), weekly_traffic, content_type

    Thresholds for issue detection (per D-01):
    - "low_links": inlinks_count is not None and inlinks_count < 3
    - "thin_content": word_count is not None and word_count < 300

    **2. `async def dispatch_batch_fix(db: AsyncSession, site_id: uuid.UUID, page_ids: list[uuid.UUID], fix_types: list[str]) -> dict`**

    The batch fix dispatches through existing audit_fix_service -> WpContentJob pipeline. The actual HTML fetching and fix generation happens inside the Celery task. For the Quick Wins dispatch, create WpContentJob records in "pending" status that the existing pipeline picks up.
