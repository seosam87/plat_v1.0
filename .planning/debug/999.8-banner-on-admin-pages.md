## Root Cause

`app/templates/base.html:100` безусловно включает `components/playbook_banner.html` на всех десктоп-страницах, а `app/static/js/playbook_banner.js` рендерит баннер для любого роута, кроме `/m/*`. Когда пользователь после нажатия «Перейти к шагу» оставляет `sessionStorage.active_playbook_step` непустым, он сохраняется до конца таб-сессии — и при переходе на админские страницы плейбук-конструктора (`/ui/playbooks/blocks`, `/ui/playbooks`, `/ui/playbooks/experts`, `/ui/playbooks/{id}/edit`) баннер появляется снова, потому что ни шаблон, ни JS не знают, что текущий роут не является «целевой» страницей шага.

Иначе говоря — Plan 05 реализовал баннер по принципу «глобальный оверлей с состоянием в sessionStorage», но забыл различение «страница, на которой шаг исполняется» vs «страница, на которой шаг настраивается». Концепция allowed_routes / blocked_routes отсутствует на обеих сторонах (template + JS).

## Options for Fix

### Option A — Чёрный список префиксов в `playbook_banner.js`

Самый минимальный фикс. Добавить в начало `init()` после mobile-проверки:

```js
// Skip on playbook admin pages — banner should only appear on action-target pages.
if (window.location.pathname.indexOf('/ui/playbooks') === 0) {
  hide();
  return;
}
```

**Плюсы:**
- 4 строки, одно изменение в одном файле
- Не трогает шаблоны, не трогает Python-роутеры
- Не ломает текущие тесты Plan 05/06
- Префикс `/ui/playbooks` — стабильный, новых функциональных «целевых» роутов под этим префиксом не планируется

**Минусы:**
- Чёрный список — хрупкий: если позже появится новая админская секция (например, `/ui/automation/`), её придётся вручную добавлять в blacklist
- Два источника правды о маршрутах: `ACTION_KIND_TO_ROUTE` (Python) и `playbook_banner.js` (JS)
- DOM баннера всё равно вставляется в страницу — `init()` просто ставит `hidden`. Минимальный визуальный «flash» теоретически возможен, но `<div hidden>` уже скрыт CSS-ом по умолчанию

### Option B — Вариант базового шаблона без баннера

Создать `base_admin.html` (или булев параметр в контексте) и изменить `playbook` admin views, чтобы они не включали `components/playbook_banner.html`.

**Плюсы:**
- Чистое разделение «админские страницы» vs «функциональные страницы» на уровне шаблонов
- Не вставляется лишний DOM на админских страницах вообще

**Минусы:**
- Нужно либо плодить варианты `base.html`, либо протаскивать булев флаг через контекст КАЖДОГО админского хендлера (≈10 routes в `playbooks.py`, плюс возможно `experts`)
- Высокий риск регрессии — `base.html` шарится со всеми остальными админскими роутами проекта (не только playbook). Создание `base_admin.html` потребует переноса всех общих блоков
- Над-инженерия для «минор-косметик» бага

### Option C — Серверный context-флаг от плейбук-роутеров

Каждый `/ui/playbooks/*` хендлер передаёт в шаблон `is_playbook_admin_page=True`, а `base.html` оборачивает include в `{% if not is_playbook_admin_page %}`.

**Плюсы:**
- Решение явное, диагностируемое со стороны сервера
- Легко расширять на другие админки

**Минусы:**
- Меняет ~10 хендлеров в `app/routers/playbooks.py`
- Создаёт неявный «соглашения-based» контракт: каждый новый админский хендлер должен помнить выставлять флаг
- Не решает проблему «забыли в новом хендлере» — если кто-то добавит `/ui/playbooks/templates/import` без флага, баг вернётся
- Полностью серверное решение для чистого UI-стейта (sessionStorage), что архитектурно неуклюже

---

### Option D — Allow-list по `action_kind → target_url` (точное совпадение)

Расширить `/api/playbook-step-active` (или `/api/playbook-step-route`), чтобы возвращать `target_url`. JS сравнивает `window.location.pathname` с этим `target_url` (или его префиксом без `{site_id}`), и показывает баннер ТОЛЬКО при совпадении.

**Плюсы:**
- Семантически самый правильный подход: «баннер показывает, что ЭТА страница — место исполнения активного шага»
- Один источник правды (`ACTION_KIND_TO_ROUTE` на сервере)
- Автоматически закрывает любые новые админки/новые роуты — не нужно обновлять blacklist

**Минусы:**
- Самый трудоёмкий: требует менять API-ответ, JS-логику и, возможно, добавлять fuzzy-matching (например, `/ui/sites/{uuid}/crawls/{crawl_id}` — это всё ещё «страница crawls», нужно matchить по префиксу с подменой плейсхолдеров)
- Возможны ложноотрицательные срабатывания: если юзер на `/ui/keywords/{site_id}/groups`, баннер должен показываться (это всё ещё контекст `open_keywords`), но строгое матчинг сломается
- Перерасход на минорный косметический баг

## Recommendation

**Option A — blacklist по префиксу `/ui/playbooks` в `playbook_banner.js`.**

Обоснование:
1. **Bug severity** — минорный косметический. Баннер не ломает функциональность админских страниц, не блокирует никаких действий, не вызывает ошибок. Чинить его over-engineered решением (Option C/D) — нарушение принципа пропорциональности.
2. **Минимум изменений** — 4 строки в одном файле. Нулевая регрессия для других подсистем.
3. **Префикс безопасен** — `/ui/playbooks/*` нет в `ACTION_KIND_TO_ROUTE` ни для одного `action_kind` (проверено в `app/services/playbook_service.py:703`). Никогда не появится: «функциональная» страница, на которой исполняется шаг плейбука, по дизайну не может находиться в админке плейбук-конструктора.
4. **Расширяемость** — если в будущем (Plan 07+) появится нужда в более точном маршрутизированном баннере (например, скрывать на `/ui/projects/{id}/kanban` или на `/ui/admin/*`), можно рефакторить в Option D одним отдельным небольшим планом, когда появится 2-3 кейса. Сейчас кейс ровно один.
5. **Согласовано с существующим mobile-skip** — JS уже использует именно blacklist-подход (`/m/*`). Добавление второго prefix-skip визуально и архитектурно симметрично.

**Acceptance check для фикса:**
1. Открыть активный шаг плейбука через «Перейти к шагу» → редирект на `/ui/keywords/{site_id}` → баннер виден ✓
2. Перейти в `/ui/playbooks/blocks` → баннер скрыт ✓
3. Перейти в `/ui/playbooks` → баннер скрыт ✓
4. Перейти в `/ui/playbooks/{any_id}/edit` → баннер скрыт ✓
5. Вернуться на `/ui/keywords/{site_id}` → баннер снова виден (sessionStorage не очищается) ✓
6. Mobile (`/m/*`) — баннер по-прежнему скрыт (существующая проверка) ✓
